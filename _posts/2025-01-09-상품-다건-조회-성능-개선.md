## 시나리오
현대 애플리케이션에서 높은 트래픽 환경에서도 빠르고 안정적인 응답을 제공하는 것은 필수적입니다. 특히, 실시간 데이터 조회 요청이 많은 경우, 데이터베이스 최적화는 시스템 성능에 직접적인 영향을 미칩니다.

이번 글에서는 100명이 동시에 API 요청을 보내는 시나리오를 설정하여 병목 현상을 분석하고, 인덱스를 활용한 성능 최적화 과정을 소개합니다.

### 조건

- `isSold`가 **true**인 데이터만 조회
- `name` 컬럼의 특정 문자열 포함
- `category` 컬럼의 특정 문자열 포함

## 문제 분석

### 분석 결과
1. 평균 응답 속도가 **3.9**초로 사용자 경험을 고려했을 때 매우 느린 속도입니다.
2. 표준 편차가 최악의 경우 **7.4**초, 최선의 경우 **0.2**초로 큰 편차를 보입니다.
이는 쿼리 병목 현상이 발생하고 있다는 신호입니다.

**해결 방법**
쿼리 병목 지점을 찾아내고 적절한 인덱스를 추가하여 성능을 개선해야 합니다

***

## 인덱스 적용 과정

### 1. 쿼리 병목 현상 분석 (EXPLAIN 사용)
`EXPLAIN` 명령어를 사용해 병목 현상을 분석한 결과는 다음과 같습니다
![](https://velog.velcdn.com/images/duwnstj12/post/b6ce4003-b715-49e9-9a1e-a97e141971f5/image.png)

**주요 분석 내용**
- **storeId**:
외래 키로 지정되어 있어 이미 인덱스가 적용된 상태였습니다.
`type: ref`로 적절한 조인 관계를 유지하고 있었습니다.

- **병목 지점**:
조회되는 row 수가 **31,920건**으로 많았으며, `Extra` 필드에서 **Using filesort**가 발생해 정렬 작업이 비효율적으로 이루어지고 있었습니다.

### 결과

***
### 3. 추가 인덱스 적용: updatedAt 컬럼
**이유**
- **updatedAt**은 자주 조회되는 컬럼이므로, 인덱스를 적용하면 성능이 개선될 가능성이 높다 판단했습니다.
- 현재 `EXPLAIN` 결과에서 `Using filesort`가 발생하고 있으므로, 정렬 효율을 개선할 필요가 있다 판단했습니다.

**인덱스 적용 코드**

```java
@Table(
        name = "product",
        indexes = {
                @Index(name = "idx_updatedAt", columnList = "updatedAt")
        }
)

```text
**적용 후 EXPLAIN 결과**
![](https://velog.velcdn.com/images/duwnstj12/post/5f60bb70-25b8-45d4-b8d9-f3a6b538208e/image.png)
- **Using where**: WHERE 조건이 사용되었음을 나타냅니다.
- **Backward index scan**: 내림차순 정렬이 수행됩니다.

***

### 4. 복합 인덱스 적용 : storeId + updatedAt
**문제**
- `updatedAt` 인덱스를 단독으로 적용한 경우, 외래 키인 `storeId`에 인덱스가 적용되지 않아 성능이 저하되었습니다.
**해결**
- 복합 인덱스를 `storeId`와 `updatedAt`에 적용했습니다.
이로 인해 두 조건을 효율적으로 처리할 수 있게 되었습니다.

## 인덱스 생성 시 고려할 사항
1. **카디널리티 (Cardinality)**:
    - 중복도가 낮은 컬럼일수록 인덱스 효과가 큼.
2. **선택도 (Selectivity)**:
    - 선택도가 낮을수록, 즉 조회 조건에 따라 적은 row를 반환할수록 인덱스 효과가 큼.
3. **조회 활용도**:
    - 자주 조회되는 컬럼은 인덱스 적용 시 효과가 큼.
4. **수정 빈도**:
- 수정이 적은 컬럼에 인덱스를 적용하는 것이 적합. 수정 시 인덱스도 갱신되므로 성능 저하 가능.

***

## 결론 
1. 쿼리의 병목 지점을 파악하여 적절히 인덱스를 적용한 결과, 조회 성능이 개선되었습니다.
2. 하지만 최적화만으로도 한계가 있으므로, 캐싱 전략(Redis)을 추가적으로 고려해야 합니다.
3. 인덱스 적용 시, 성능 저하 요인을 항상 함께 분석하며 신중히 설계해야 합니다.
