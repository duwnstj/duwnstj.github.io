## Amazon S3
- **S3**은 **제한 없는 확장성**을 제공하는 스토리지 서비스로 백업,저장소,재해 복구, 아카이브, 하이브리드 클라우드 스토리지 등 다양한 유스 케이스에 활용됩니다.
- 빅데이터 분석을 위한 **데이터 레이크나 정적 웹사이트 호스팅,소프트웨어 업데이트** 등에 자주 사용됩니다.

## S3의 주요 용어
- **버킷** : 파일을 저장하는 최상위 디렉토리 역할을 하며 **전역적으로 고유한 이름**을 가져야합니다. 버킷은 특정 **리전** 내에서 생성되고 관리됩니다.
- **오브젝트** : S3에 저장되는 파일로 각 오브젝트는 **키(key)**를 가집니다 키는 해당 오브젝트의 전체 경로를 나타내며 S3에서는 디렉토리 개념 대신 키를 기반으로 오브젝트를 관리합니다.

## S3의 주요 기능
- **오브젝트 스토리지** : S3에서는 최대 5TB 크기의 파일을 업로드할 수 있으며 대용량 파일은 **멀티파트 업로드** 기능을 사용해 여러 조각으로 나눠 업로드 가능합니다.
- **메타데이터와 태그** : 각 오브젝트는 **메타데이터**를 가지며 사용자나 시스템이 키-값 쌍으로 메타데이터를 설정할 수 있습니다. 또한 태그를 통해 보안 및 관리에 유용한 정보를 저장할 수 있습니다.
- **버전 관리** : 버전 관리를 활성화하면 오브젝트에 **버전 ID**가 부여되어 변겅 내역을 추적할 수 있습니다. 

## Amazon S3의 보안

### 1. IAM 정책
- **IAM 정책**은 특정 **사용자가 어떤 API**를 호출할 수 있는지 정의하는 보안 설정입니다. 사용자는 **IAM 정책**에 따라 S3 버킷 및 오브젝트에 접근할 수 있습니다.

### 2. S3 버킷 정책
- **S3 버킷 정책은 리소스** 단위로 설정할 수 있는 보안 정책입니다. 버킷 정책을 통해 **특정 사용자나 다른 계정**에서 S3 버킷에 접근을 허용하거나 차단할 수 있습니다.
- **교차 계정 액세스**도 지원되며 다른 계정 사용자가 S3버킷에 접근할 수 있도록 허용할 수 있습니다.

### 3. 액세스 제어 목록
- **ACL**을 사용하면 오브젝트나 버킷에 대한 세부적인 접근 권한을 설정할 수 있습니다. 그러나 **버킷 **ACL**은 많이 사용되지 않으며 비활성화할수 있습니다.

### 4. 암호화 및 보안 적용
- **암호화** : S3 오브젝트는 암호화할 수 있으며 이를 통해 데이터가 더욱 안전하게 보호됩니다.
- **퍼블릭 액세스 차단** : AWS는 **퍼블릭 액세스를 버킷 수준**에서 차단하는 기능을 제공하여 기업 데이터의 유출을 방지합니다. 잘못된 설정으로 인해 버킷이 공개되지 않도록 보호하는 역할을 합니다.

### 5. 정책 설정 예시 
- **정책의 구조**는 JSON 형식으로 작성되며 **Resource,Effect(허용 또는 거절),Action(허용하는 API), Principal(대상 사용자)**를 정의합니다.
- 예를 들어 **GetObject** 액션을 허용하는 정책을 설정하면 특정 버킷 내의 오브젝트에 대해 누구나 접근할 수 있도록 설정할 수 있습니다.

### 6. IAM 역할과 EC2 인스턴스
- **EC2 인스턴스**가 S3 버킷에 접근해야할 경우 **IAM 역할**을 부여하여 EC2 인스턴스에서 S3에 안전하게 접근할 수 있습니다.

### 7. 퍼블릭 액세스 차단 설정
- 버킷이 **공개적으로 접근**되지않도록 AWS가 제공하는 **퍼블릭 액세스 차단** 기능을 활성화 할 수 있으며 **계정 단위**에서도 이 설정을 적용할 수 있습니다. 이는 데이터 유출을 방지하기 위한 필수 설정입니다. 


## S3 버킷으로 웹사이트 호스팅
- **정적 웹사이트** : S3는 HTML, 이미지,CSS 파일과 같은 **정적 파일**을 호스팅할 수 있습니다. 동적 웹사이트가 아닌 **정적 콘텐츠**를 제공하는 경우에 적합합니다.
- **URL 형식** : 웹사이트의 URL은 **리전**에 따라 형식이 약간 다릅니다. 일반적으로 `bucket-name.s3-website.ap-northeast-2.amazonaws.com` 또는 `bucket-name.s3-website-ap-northeast-2.amazonaws.com`와 같은 형태로 제공됩니다.

## 퍼블릭 액세스 설정
- **퍼블릭 읽기 권한** : 웹사이트가 인터넷에서 접근 가능하려면 **S3 버킷에 퍼블릭 읽기 권한**이 설정되어야합니다.
- **S3 버킷 정책** : 버킷 정책을 통해 **퍼블릭 읽기**를 활성화하여 누구나 웹사이트 파일에 접근 할 수 있도록 설정해야합니다. 만약 설정이 잘못되어 있으면 **403 Forbidden** 오류가 발생할 수 있습니다. 이 경우 퍼블릭 접근을 허용하는 **S3 버킷 정책**을 추가해야합니다. 

## Amazon S3 버전 관리 기능
Amazon S3의 버전 관리 기능은 파일을 안전하게 관리하고 의도치 않은 삭제나 수정으로부터 데이터를 보호하는 유용한 도구입니다. 버전 관리 기능은 **S3 버킷** 단위에서 설정되며 활성화 하면 파일을 업데이트하거나 삭제할 때마다 **새 버전**이 생성됩니다.

### 1. 버전 관리 기능의 핵심 특징
- **새 버전 생성** : 버킷에서 버전 관리 기능이 활성화되면 같은 키로 파일을 업로드 할 때마다 새 버전이 생성됩니다. 예를 들어 동일한 파일을 다시 업로드 할 경우 이전 버전은 유지되고 새로운 파일은 새로운 **버전 번호**로 저장됩니다.
- **삭제 복구** : 파일을 삭제하더라도 실제로 삭제되지 않고 `삭제 마커`가 추가되므로 이전 버전으로 **롤백**할 수 있습니다. 이를 통해 의도치 않은 파일 삭제를 **복구**할 수 있습니다.
- **이전 버전 롤백** : 원하는 시점으로 파일을 되돌릴 수 있습니다. 예를 들어, 이틀 전에 업로드 한 파일로 **되돌리기**하여 데이터의 이전 상태를 쉽게 복원할 수 있습니다.

### 2. 유의 사항
- **버전 값** : 버전 관리 기능을 활성화 하기 전의 파일들은 **null 버전**으로 저장됩니다. 버전 관리 기능을 활성화하거나 비활성화해도 기존에 저장된 파일들은 계속 유지됩니다.
- **중단 후 관리** : 버전 관리 기능을 중단하더라도 이미 생성된 버전은 삭제되지 않고 안전하게 남아있습니다. 

## Amazon S3 복제 기능
Amazon S3는 데이터를 복제하는 두가지 방법을 제공합니다. **리전 간 복제(CRR)와 동일 리전 복제(SRR)**입니다. 이 기능들은 데이터를 다른 버킷으로 비동기적으로 복사하여 데이터 보호 및 관리의 유연성을 높여줍니다.

### 1. 리전 간 복제(CRR)
-**CRR은 두 리전 간**에 데이터를 복제하는 방식입니다. 예를 들어 한 리전에 있는 버킷에서 다른 리전의 버킷으로 데이터를 비동기적으로 복제할 수 있습니다.
- CRR의 유용한 사례는 **컴플라이언스 요건**을 충족시키거나 **재해 복구**를 목적으로 다른 리전에 데이터를 복제하는 상황입니다. 또한 **대기 시간을 최소화하고, 다른 계정 간의 데이터 복제**가 필요할 때도 유용합니다.

### 2. 동일 리전 복제(SRR)
- **SRR은 같은 리전 내**에서 데이터를 복제하는 방식입니다. 같은 리전 내에서 여러 버킷 사이에 데이터를 비동기적으로 복사할 수 있습니다.
- SRR은 **로그집계나 운영 계정과 테스트 계정** 간에 데이터를 실시간으로 복제할 때 유용합니다. 이는 **테스트 환경**이 따로 있는 경우 특히 도움이 됩니다.

### 3. 복제 작업 설정
- **버전 관리 활성화** : 복제를 시작하기 전에 소스 버킷과 대상 버킷에 **버전 관리** 기능을 활성화해야합니다.
- **권한 설정** : 복제 작업을 실행하기 위해서는 S3 서비스에 적절한 **IAM 권한**을 부여해야합니다. 특히 소스 버킷에는 **읽기 권한**, 대상 버킷에는 **쓰기 권한**이 필요합니다. 

## AmazonS3 스토리지 클래스

### 1. Amazon S3 Standard
- 내구성 : 99.99999999999%(9가 11개)
- 가용성 : 99.99%
- 사용 용도 : 자주 액세스 하는 데이터를 위한 기본 스토리지입니다. 빅데이터 분석,모바일 및 게임 애플리케이션, 콘텐츠 배포에 주로 사용됩니다. 가장 일반적인 사용 사례에 적합합니다.

### 2. S3 Standard-Infrequent Access
- 내구성 : 99.999999999%
- 가용성 : 99.9%
- 사용용도 : 자주 접근하지 않지만 필요할 때 빠르게 접근해야하는 데이터를 위한 스토리지입니다. 재해 복구,백업과 같은 비상시 활용되는 데이터에 적합하며 비용 절감을 위해 설계되었습니다.

### 3. S3 One Zone-IA
- 내구성 : 99.999999999%(단일 AZ에서 제공)
- 가용성 : 99.5%
- 사용 용도 : 단일 가용 영역에 데이터를 저장하기 때문에 리스크가 있을 수 있지만 비용 절감이 큰 장점입니다. 주로 온프레미스 백업의 보조 복사본 저장에 적합합니다.

### 4. Amazon S3 Glacier
- 사용용도 : 장기 데이터 아카이브 및 콜드 스토리지에 적합한 스토리지입니다. 비용이 저렴하고 다양한 검색 옵션을 제공합니다.
- 종류 : 
    - Glacier Instant Retrieval : 밀리초 단위로 데이터에 즉시 접근할 수 있는 옵션입니다. 주기적으로 접근하는 백업에 적합합니다.
    - Glacier Flexible Retrieval : 빠르게는 1분, 느리게는 12시간 내에 데이터를 복구할 수 있는 옵션으로 다양한 검색 속도를 제공합니다.
    - Glacier Deep Archive : 12~48시간의 검색 시간을 제공하며 비용이 가장 저렴한 장기 보관용 스토리지입니다.
    
### 5. S3 Intelligent-Tiering
- **사용 용도** : 데이터 접근 패턴을 자동으로 분석하여 자주 사용되지 않는 데이터를 저렴한 스토리지 계층으로 이동시키는 스토리지입니다. 모니터링과 자동화 요금이 발생하지만 장기적으로 비용을 절약할 수 있습니다.
- **장점** : 사용자가 직접 데이터를 관리하지 않아도 되는 편리함을 제공하며 검색 요금이 발생하지 않습니다.

### 6. 내구성 및 가용성
- **내구성** : 데이터 손실 가능성을 나타내며 모든 스토리지 클래스에서 동일한 99.999999999%의 내구성을 제공합니다. 이는 S3에 데이터를 저장했을때 거의 손실되지않는다는 의미입니다.
- **가용성** : 각 스토리지 클래스마다 다르며 S3 Standard는 99.99%, One Zone-IA는 99.5%의 가용성을 제공합니다. 가용성은 데이터에 얼마나 쉽게 접근할 수 있는지를 나타냅니다. 

### 7. 비용

- 스토리지 클래스에 따라 저장 비용이 다릅니다. 자주 접근하는 데이터는 비용이 높고, 장기 보관 데이터는 저렴한 스토리지 클래스를 선택하는 것이 좋습니다. 적절한 스토리지 클래스를 선택하면 비용을 효과적으로 절감할 수 있습니다.
