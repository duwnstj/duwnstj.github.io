## 들어가며

AWS를 처음 쓰면 기본 제공되는 Default VPC를 그냥 사용하게 된다.
그런데 Default VPC는 모든 리소스가 퍼블릭으로 열려있어서 DB까지 인터넷에 노출된다.
실무에서는 절대 Default VPC를 쓰지 않는다.

이번 글에서는 **퍼블릭/프라이빗 서브넷을 직접 분리해서 VPC를 설계하는 방법**을 정리한다.

---

## VPC가 뭔가

VPC(Virtual Private Cloud)는 AWS 안에 내가 직접 만드는 격리된 가상 네트워크다.

쉽게 비유하면 이렇다.

```bash
AWS 전체     = 거대한 아파트 단지
VPC          = 내가 임대한 한 동
서브넷       = 그 동의 각 층
EC2, RDS     = 각 층에 있는 방
보안그룹     = 각 방의 도어락
```
---

## 왜 퍼블릭/프라이빗을 나누는가

핵심은 **DB를 인터넷에서 직접 접근하지 못하게 숨기는 것**이다.

```bash
인터넷
  ↓
[퍼블릭 서브넷]   ← 외부에서 접근 가능
  EC2 (웹서버)
  ALB (로드밸런서)

[프라이빗 서브넷]  ← 외부에서 직접 접근 불가
  RDS (DB)`

EC2는 외부에서 접근받아야 하니 퍼블릭에, DB는 EC2를 통해서만 접근되어야 하니 프라이빗에 배치한다.

```
---

## CIDR 이해하기

VPC와 서브넷을 만들 때 CIDR 표기법으로 IP 범위를 지정한다.

```text
10.0.0.0/16  →  10.0.0.0 ~ 10.0.255.255  (65,536개 IP)
10.0.1.0/24  →  10.0.1.0 ~ 10.0.1.255    (256개 IP)

뒤 숫자(prefix)가 클수록 범위가 좁아진다
/16 > /24 > /28
```
---

## 실습 구성

이번에 만든 구조는 다음과 같다.

```text
VPC: 10.0.0.0/16

퍼블릭 서브넷 A:   10.0.1.0/24  (ap-northeast-2a)
퍼블릭 서브넷 B:   10.0.2.0/24  (ap-northeast-2c)

프라이빗 서브넷 A: 10.0.3.0/24  (ap-northeast-2a)
프라이빗 서브넷 B: 10.0.4.0/24  (ap-northeast-2c)`

```
서브넷을 가용영역(AZ)별로 2개씩 만드는 이유는 **이중화** 때문이다.
한 데이터센터에 장애가 생겨도 다른 AZ가 서비스를 이어받는다.

---

## Step 1. VPC 생성

AWS 콘솔 → VPC → VPC 생성

`이름 태그: my-vpc
IPv4 CIDR: 10.0.0.0/16`

---

## Step 2. 서브넷 4개 생성

VPC → 서브넷 → 서브넷 생성

| 이름 | CIDR | 가용영역 |
| --- | --- | --- |
| public-subnet-a | 10.0.1.0/24 | ap-northeast-2a |
| public-subnet-b | 10.0.2.0/24 | ap-northeast-2c |
| private-subnet-a | 10.0.3.0/24 | ap-northeast-2a |
| private-subnet-b | 10.0.4.0/24 | ap-northeast-2c |

---

## Step 3. 인터넷 게이트웨이 생성 및 VPC 연결

인터넷 게이트웨이(IGW)는 퍼블릭 서브넷이 인터넷과 통신할 수 있는 문이다.
IGW가 없으면 EC2를 만들어도 외부에서 접근이 불가능하다.

```VPC → 인터넷 게이트웨이 → 인터넷 게이트웨이 생성
이름: my-igw

생성 후 → 작업 → VPC에 연결 → my-vpc 선택
```
상태가 `detached` → `attached`로 바뀌면 성공이다.

---

## Step 4. 라우팅 테이블 설정

IGW를 만들었다고 바로 인터넷이 연결되는 게 아니다.
**"퍼블릭 서브넷의 트래픽은 IGW로 내보내라"** 는 라우팅 규칙을 따로 설정해야 한다.

```VPC → 라우팅 테이블 → 라우팅 테이블 생성
이름: public-rt
VPC: my-vpc

라우팅 탭 → 라우팅 편집
0.0.0.0/0 → my-igw 추가

서브넷 연결 탭 → 서브넷 연결 편집
public-subnet-a, public-subnet-b 선택
```
프라이빗 서브넷은 별도 라우팅 테이블을 만들지 않아도 된다.
기본 라우팅 테이블(인터넷 연결 없음)이 자동으로 적용된다.

---

## 인터넷 게이트웨이 vs NAT 게이트웨이

|  | 인터넷 게이트웨이 | NAT 게이트웨이 |
| --- | --- | --- |
| 방향 | 양방향 (외부 ↔ 내부) | 단방향 (내부 → 외부) |
| 사용 위치 | 퍼블릭 서브넷 | 프라이빗 서브넷 |
| 사용 목적 | EC2 외부 접근 허용 | DB 서버의 패키지 설치 등 |
| 비용 | 무료 | 유료 (시간당 과금) |

NAT 게이트웨이는 프라이빗 서브넷에서 외부로 나가는 트래픽만 허용하고, 외부에서 직접 접근은 막는다.

---

## 최종 구성

```        인터넷
          ↓
   [인터넷 게이트웨이]
          ↓
  [퍼블릭 라우팅 테이블]
    ↙           ↘
[public-a]   [public-b]
 2a AZ          2c AZ
    
[private-a]  [private-b]  ← 인터넷 직접 연결 없음
 2a AZ          2c AZ
```
