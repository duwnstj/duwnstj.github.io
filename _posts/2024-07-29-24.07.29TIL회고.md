 # 오늘의 첫번째 알고리즘 문제
[가운데 글자 가져오기](https://velog.io/@duwnstj12/%EA%B0%80%EC%9A%B4%EB%8D%B0-%EA%B8%80%EC%9E%90-%EA%B0%80%EC%A0%B8%EC%98%A4%EA%B8%B0)

## 풀이과정

- 오늘 가운데 글자 가져오기라는 문제를 풀 때 어려운 점은 없었다.

- 하지만 첫번째 풀이에서 변수를 사용하지 않아 가독성이 떨어지고, 불필요한 `for문`을 사용했다는 것을 코드 분석 후 알게 되었다.

## 개선 과정

- `s.length()/2` 값을 변수에 담아 가독성을 개선했다. 
- 또 `substring()`메서드를 사용하였는데 for문 사용은 불필요하다 생각이 들어 for문을 제거했다.

## 느낀점
- 알고리즘 코드를 짜고 나서 기술블로그에 작성을 위해서 코드 분석을 해보니 이렇게 코드를 개선할 점을 찾게 되는것을 보고 한번더 코드 리뷰를 하는 습관이 중요하다는 것을 알았다.
- 또 기술블로그를 작성하는것이 중요하다는것을 다시 한번 더 깨닫게 되었다. 그냥 코드를 푼것으로 만족하고 다음 문제로 넘어 갔더라면 이렇게 코드의 가독성을 위해 `변수 선언`을 해야겠다는 생각과 `for문` 제거를 생각 못했기 때문이다.

 ## 오늘의 두번째 알고리즘 문제
[제일 작은 수 제거하기](https://velog.io/@duwnstj12/%EC%A0%9C%EC%9D%BC-%EC%9E%91%EC%9D%80-%EC%88%98-%EC%A0%9C%EA%B1%B0%ED%95%98%EA%B8%B0)

## 풀이과정

- `Math.min()`메서드에 대해서 알게 되었다. `Math.min(a,b)`메서드는 a와 b를 비교하여 더 작은 값을 반환해주는 매서드이다.
- 테스트에서는 성공했지만 제출 후 런타임 오류가 났다. 

## 문제 해결 과정
- 에러 원인은 answer 배열보다 `for문`의 반복 횟수가 더 많았기 때문이다.
- arr 배열에서 제일 작은 값을 제외한 값을 저장하는 answer 배열의 인덱스를 잘못 사용한 것이 문제였다.
- count 변수를 새로 만들어 문제를 해결했다.

## 느낀점
- 이 문제를 푸면서 느낀점은 테스트에서 성공해도 제출했을 때 그 코드에 에러가 있다면 런타임 오류가 뜰 수 있다는 것이였다. 
- 코드 작성 시 배열의 크기와 인덱스 사용에 주의해야 한다는 점을 배웠다.
