## 1. 개요 및 아키텍처 진화 (AS-IS vs TO-BE)

기존 로컬 개발 환경에서는 Spring Boot(WAS)의 `8080` 포트를 외부에 그대로 노출하여 가동했습니다. 하지만 이는 실제 운영(Production) 환경에서는 인프라가 무방비하게 공격에 노출되는 심각한 보안 취약점을 유발합니다.

따라서 앞단에 Nginx를 강력한 수문장(Reverse Proxy)으로 세우고, 외부에서는 절대 접근할 수 없는 독립된 도커 가상 네트워크망(`internal`)을 구축하여 네트워크 레이어를 안전하게 격리했습니다.

- **AS-IS (레거시):** 외부 사용자 ➡️ Spring Boot (`8080` 포트 다이렉트 노출)
- **TO-BE (고도화):** 외부 사용자 ➡️ **Nginx (`80`)** ➡️ 도커 가상 내부망(`internal`) ➡️ Spring Boot (`8080` 외부 차단 및 은닉)

``` bash
## 3. 원인 분석 및 치열했던 3대 '사고 검증'

에러를 해결하기 위해 무작정 구글링이나 커뮤니티를 뒤지기 전, 인프라의 동작 메커니즘을 기반으로 가설을 세우고 치열하게 검증하는 프로세스를 거쳤습니다.

### 💡 [사고 검증 1] `localhost` 주소 해석의 한계 추적

- **가설:** 도커 가상 네트워크 내부에서 `localhost:3306`으로 DB 접속을 시도하면 컨테이너들은 어떻게 움직일까?
- **결론:** 컨테이너 세상 속에서 `localhost`는 '자기 자신(해당 컨테이너 방구석)'을 의미한다. `app` 컨테이너가 자기 자신의 방에 대고 3306 포트를 열어달라고 외치니 당연히 `Connection refused`가 터진 것이다. 도커 브릿지 가상망 안에서 컨테이너끼리 통신하려면 주소창에 `localhost`가 아닌 컴포즈 서비스명인 `db`를 명시해야 라우팅이 정상적으로 흘러간다.

### 💡 [사고 검증 2] `ddl-auto: validate`의 엄격함 이해

- **가설:** 로컬 개발계에서는 편하게 `update` 설정을 쓰는데, 운영계(`prod`)에서 이를 분리해 `validate`를 거는 본질적인 이유는 무엇일까?
- **결론:** `validate` 옵션은 실제 DB 구조를 자바 코드가 절대로 건드리지 못하게 꽁꽁 묶고 '구조 검증'만 한다. 만약 배포 시점에 자바 엔티티와 운영 DB 테이블 구조가 단 1글자라도 다르면 예외를 터트리며 **서버를 즉사(기동 실패)** 시킨다. 실시간 고객 데이터 유실 및 잘못된 SQL 실행을 막기 위한 프로덕션 환경의 철저한 방어벽 메커니즘이다. 따라서 실무에서는 [DB 컬럼 선행 반영 쿼리 실행] 후 [애플리케이션 소스 코드 배포]라는 타이밍 분리가 필수적이다.

### 💡 [사고 검증 3] 헬스체크 엔드포인트의 환경적 엇박자

- **가설:** 도커 엔진이 10초마다 찔러보는 주소와 실제 자바 경량 리눅스 환경의 삼중 대조가 맞물려 있는가?
- **결론:** `Dockerfile`에서 빌드한 최종 베이스 이미지는 매우 슬림한 경량형 `Alpine JRE` 리눅스이다. 이 미니멀한 OS 안에는 기본적으로 대중적인 `curl` 도구가 내장되어 있지 않아서 헬스체크가 실패했던 것이다. 또한, 자바 컨트롤러 클래스의 `@RequestMapping` 경로와 컴포즈 파일에 명시한 `wget` 타겟 URL 주소의 미세한 싱크가 맞지 않아 도커 엔진이 응답을 받지 못했다.

## 4. 해결 및 최종 최적화 코드

분석한 가설을 바탕으로 1) 컴포즈 내부 DB 호스트를 서비스명(`db`)으로 강제 치환하고, 2) 헬스체크 명령어를 Alpine 내장형인 `wget`으로 교체 및 컨트롤러 실제 URL 경로와 100% 동기화하여 완벽한 인프라 뼈대를 완성했습니다.

### 💻 `docker-compose.yml` (로컬 개발용 최종 마스터피스)



{% endhighlight %}YAML
# [주석] 최신 도커 엔진 스펙 스탠다드에 맞춰 경고 로그를 유발하는 최상단 'version' 속성은 과감히 생략했습니다.

services:
  db:
    image: mysql:8.0
    container_name: cover-mysql
    environment:
      MYSQL_ROOT_PASSWORD: ${SPRING_DATASOURCE_PASSWORD} # .env에 격리한 로컬 DB 패스워드 주입
      MYSQL_DATABASE: cover_db
    networks:
      - internal # [핵심] 외부 접근이 차단된 격리된 내부 가상 네트워크망에 합류
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost", "-u", "root", "-p$${SPRING_DATASOURCE_PASSWORD}"]
      interval: 5s
      timeout: 5s
      retries: 5

  app:
    build: .
    container_name: cover-app
    env_file:
      - .env # 로컬 환경 변수 주입
    environment:
      SPRING_PROFILES_ACTIVE: ${SPRING_PROFILES_ACTIVE}
      # [해결] .env의 localhost를 오버라이딩하여 도커 내부망 통신을 위해 호스트 주소를 'db' 서비스명으로 정밀 세팅
      SPRING_DATASOURCE_URL: jdbc:mysql://db:3306/cover_db?serverTimezone=Asia/Seoul
      SPRING_DATASOURCE_USERNAME: ${SPRING_DATASOURCE_USERNAME}
      SPRING_DATASOURCE_PASSWORD: ${SPRING_DATASOURCE_PASSWORD}
    networks:
      - internal # db 및 nginx와 동일한 가상망 브릿지로 연결
    depends_on:
      db:
        condition: service_healthy # DB 컨테이너가 헬스체크 통과(Healthy) 판정을 받을 때까지 자바 앱 기동을 영리하게 대기시킴
    healthcheck:
      # [해결] Alpine 리눅스 호환을 위해 wget 명령어로 선회하고 실제 스프링 컨트롤러 주소와 완벽 매칭
      test: ["CMD", "wget", "--no-verbose", "--tries=1", "--spider", "http://localhost:8080/api/health"]
      interval: 10s
      timeout: 5s
      retries: 5

  nginx:
    image: nginx:alpine
    container_name: cover-nginx
    ports:
      - "80:80" # 외부 클라이언트는 오직 표준 웹 포트(80)로만 접근 가능. 백엔드의 8080 포트는 외부에서 철저히 은닉됨
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro # 호스트의 Nginx 프록시 설정을 컨테이너 내부로 읽기 전용(:ro) 마운트
    depends_on:
      app:
        condition: service_healthy # 스프링 부트 서버가 완전히 부팅되어 200 OK(Healthy) 사인을 켜기 전까지 Nginx 출격 대기
    networks:
      - internal # 수신한 웹 요청을 가상 내부망을 통해 app 컨테이너로 안전하게 토스

networks:
  internal:
    driver: bridge # 도커 컴포즈 내부에서 컨테이너 간 가상 독립 IP 라우팅을 제어하는 독립 브릿지 드라이버 선언
```
