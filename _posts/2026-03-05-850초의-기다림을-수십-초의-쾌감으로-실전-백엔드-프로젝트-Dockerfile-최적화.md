기존에 팀원들과 함께 개발하던 **Spring Boot 멀티 모듈 프로젝트**에 도커를 입히며 겪은 시행착오를 기록합니다. 특히 `module-owner` 모듈을 대상으로 이미지 용량과 빌드 속도를 개선하며 얻은 엔지니어링 통찰을 공유합니다.

## **📊 1. 최적화 결과 요약: "데이터로 증명하다"**

실습 결과, 이미지 용량과 빌드 시간에서 압도적인 성능 차이를 확인했습니다.

| 구분 | DockerFile.bad (기존) | DockerFile.good (개선) | 개선 결과 |
| --- | --- | --- | --- |
| **이미지 용량** | **1.34 GB** | **522 MB** | **약 60% 다이어트 성공** |
| **최초 빌드 시간** | 850 초 | 700 초 | 환경 이동 및 첫 투자 비용 |
| **재빌드 시간** | 850 초 (반복) | **단 수십 초** | **레이어 캐싱의 승리** |

## **🛑 2. 트러블슈팅: 백엔드 적용 중 마주한 벽**

### **📍 멀티 모듈 프로젝트의 특수성**

- **상황**: 전체 프로젝트가 아닌 특정 모듈(`module-owner`)만 빌드하고 배포해야 하는 상황.
- **해결**: `WORKDIR`와 `COPY` 경로를 정밀하게 설정하여 필요한 모듈의 빌드 결과물(`.jar`)만 추출하도록 설계했습니다.

### **📍 OS 및 도커 환경 충돌**

- **문제**: 윈도우(WSL)와 VM 간의 가상화 충돌로 로컬 빌드 불가.
- **해결**: GitHub을 통해 프로젝트 코드를 VM 리눅스로 이동하여 환경 독립성을 확보했습니다.

## **🛠️ 3. 기술 분석: Bad vs Good 무엇이 다른가?**

### **❌ DockerFile.bad (비효율의 결정체)**

- **단일 스테이지**: JDK(개발도구)와 빌드 라이브러리가 최종 이미지에 그대로 포함되어 용량이 비대함.
- **보안 취약**: ROOT 계정 사용으로 컨테이너 탈취 시 호스트 리소스 노출 위험.
- **캐시 붕괴**: `COPY . .`을 상단에 배치하여, 코드 한 줄만 바꿔도 850초의 빌드 과정을 무한 반복.

### **✅ DockerFile.good (운영 환경의 정석)**

- **멀티 스테이지 빌드**: 빌드(JDK)와 실행(JRE) 단계를 분리하여 실행에 필요한 `app.jar`만 남김.
- **보안 계정 적용**: `useradd`로 일반 유저(`appuser`)를 생성하여 최소 권한 원칙 준수.
- **전략적 레이어 분리**: 설정 파일(Gradle 관련)을 소스 코드보다 먼저 복사하여 재빌드 속도 극대화.

---

### 3-1 Dockerfile 파일명 규칙과 -f 옵션

Docker는 기본적으로 `Dockerfile`이라는 정확한 파일명만 인식합니다. 대문자 D로 시작하고, 확장자 없이 딱 이 이름이어야 합니다.

그런데 실무에서는 환경별로 다른 Dockerfile이 필요할 때가 있습니다. 개발 환경에서는 디버깅 도구를 포함하고, 운영 환경에서는 최대한 경량화하는 식입니다.

| 파일명 | 용도 |
| --- | --- |
| Dockerfile.dev | 개발 환경 |
| Dockerfile.prod | 운영 환경 |
| Dockerfile.test | 테스트 환경 |

이렇게 파일명을 다르게 하면 `-f` 옵션으로 어떤 파일을 빌드할지 지정합니다.



``` bash
# 기본 Dockerfile → 옵션 불필요
docker build -t myapp .
```
### 3-2 다른 파일명 → -f 옵션 필수
`docker build -f Dockerfile.dev -t myapp:dev .`

### 3-3 다른 경로에 있는 경우
`docker build -f ./docker/Dockerfile.prod -t myapp:prod .`

- `f` 옵션 없이 `Dockerfile.dev`를 빌드하려고 하면 "unable to find Dockerfile" 에러가 발생합니다. Docker가 기본 파일명인 `Dockerfile`만 찾기 때문입니다.

## **🔍 4. 명령어 딥다이브: 핵심 원리 이해**

- **`WORKDIR`**: 리눅스의 `cd` 역할. 폴더가 없으면 자동 생성하며, 경로를 고정하지 않으면 ROOT 경로가 혼잡해져 유지보수가 힘들어집니다.
- **`ADD` vs `COPY`**: 단순 파일 복사는 `COPY`, URL 인식이나 압축 해제가 필요할 때만 `ADD`를 권장합니다.
- **`&&` (AND 연산자)**: 명령어를 하나로 묶어 **레이어 개수(최대 127개)**를 줄이고 이미지 효율을 높입니다.

---

## **🧠 5. 핵심 통찰: 700초와 "CACHED"의 비밀**

### **⚡ 왜 Bad는 매번 850초가 걸렸을까?**

`bad` 파일은 모든 파일을 한꺼번에 복사(`COPY . .`)합니다. 이 경우 소스 코드를 수정하면 도커는 **"파일 덩어리가 변했다"**고 판단하여 그 아래의 모든 캐시를 파괴합니다. 결국 700초가 넘는 라이브러리 다운로드를 매번 처음부터 다시 하는 '도미노 현상'이 발생합니다.

### **⚡ "진짜 수십 초면 되네?" - 레이어 분리의 힘**

`good` 방식의 핵심은 **순서**입니다. 변하지 않는 설정 파일(`build.gradle` 등)을 먼저 복사해 라이브러리 레이어를 미리 구워두면, 소스 코드가 변해도 이미 다운로드된 라이브러리는 `CACHED` 상태로 재사용됩니다. 덕분에 700초의 과정을 건너뛰고 컴파일만 수행하여 단 수십 초 만에 빌드가 끝납니다.

## 6. `.dockerignore` 사용하기

Dockerfile을 이용해 이미지를 빌드할 때 중요한 점이 있습니다.

개발을 할 때 보통 **민감한 정보(API Key, Secret Key, DB 비밀번호 등)**를 `.env` 파일에 저장하여 **환경변수로 관리**합니다. 만약 이러한 파일이 Docker 이미지를 만들 때 함께 포함된다면, 이미지가 외부에 공유되거나 저장소에 올라갈 경우 **민감한 정보가 노출될 위험**이 있습니다. 이는 보안적으로 매우 위험한 상황이 될 수 있습니다.

이러한 문제를 방지하기 위해 Docker에서는 `.gitignore`와 유사한 개념인 **`.dockerignore`** 파일을 제공합니다.

`.dockerignore`의 역할은 **Docker 이미지 빌드 시 Docker가 사용하는 빌드 컨텍스트(Build Context)에 특정 파일이나 디렉토리를 포함하지 않도록 제외하는 것**입니다. 즉, `.dockerignore`에 명시된 파일들은 **이미지 빌드 과정에서 Docker에게 전달되지 않기 때문에 이미지에 포함되지 않습니다.**

이를 통해 다음과 같은 장점이 있습니다.

- 민감한 정보가 이미지에 포함되는 것을 방지 (보안)
- 불필요한 파일을 제외하여 이미지 크기 감소
- 빌드 컨텍스트 크기를 줄여 Docker 빌드 속도 향상

일반적으로 `.dockerignore`에는 다음과 같은 파일들을 추가합니다.

```text
.env
.git
.gitignore
node_modules
Dockerfile
README.md
```
예를 들어 `.env` 파일을 `.dockerignore`에 추가하면, Docker 이미지를 빌드할 때 해당 파일이 포함되지 않아 **환경 변수 정보가 이미지 내부에 노출되는 것을 방지할 수 있습니다.**

따라서 Docker 이미지를 안전하고 효율적으로 관리하기 위해서는 `.dockerignore` 파일을 반드시 설정하여 **민감한 정보나 불필요한 파일이 이미지에 포함되지 않도록 관리하는 것이 중요합니다.**

## **💡 7. 최종 요약: 엔지니어를 위한 4가지 철칙**

1. **멀티 스테이지 필수**: 빌드 도구와 실행 환경을 분리하여 이미지를 가볍게 유지하세요.
2. **최소 권한 원칙**: 보안사고 피해 최소화를 위해 항상 일반 사용자로 컨테이너를 구동하세요.
3. **레이어 설계**: "변화가 적은 설정을 위로, 자주 변하는 코드를 아래로" 배치해 캐시 효율을 극대화하세요.
4. **권한 명시**: 리눅스 환경 이동 시 `chmod +x gradlew` 등으로 실행 권한을 확실히 부여하세요.
