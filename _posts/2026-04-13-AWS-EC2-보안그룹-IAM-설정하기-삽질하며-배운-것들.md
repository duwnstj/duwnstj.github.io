## 들어가며

저번 글에서 VPC를 직접 설계했다. 이번엔 그 VPC 위에 실제 서버(EC2)를 올리는 과정을 정리한다.
따라하면서 삽질을 꽤 했는데, 그 덕분에 개념이 훨씬 명확하게 잡혔다.

---

## 오늘 만들 것

```bash 
인터넷
  ↓
[보안그룹]   ← EC2 앞에 세우는 방화벽
  ↓
[EC2]        ← 퍼블릭 서브넷에 올라가는 서버
  ↓
[IAM 역할]   ← EC2가 AWS 서비스 접근할 때 쓰는 신분증
```text
---

## 개념 정리

### 보안그룹이 뭔가

EC2 앞에 세워두는 방화벽이다.

```bash
"22번 포트(SSH)는 내 IP만 허용"
"8080번 포트(Spring Boot)는 모든 IP 허용"
"그 외 모든 포트는 차단"
```text
이런 규칙을 정하는 것이다.

### IAM 역할이 뭔가

EC2가 다른 AWS 서비스에 접근할 때 필요한 신분증이다.

```bash
EC2 위 Spring Boot가 S3에 파일 올리려면
→ EC2한테 "S3 써도 돼" 권한을 줘야 함
→ 그게 IAM 역할(Role)
```text
여기서 중요한 것이 있다.

```text
IAM 사용자 (User) → 사람이 AWS 콘솔/CLI 쓸 때
IAM 역할   (Role) → AWS 서비스끼리 권한 줄 때
```text
처음에 IAM 역할을 만들어야 하는데 IAM 사용자를 만드는 실수를 했다.
EC2에 붙이는 건 반드시 **IAM 역할**이어야 한다.

---

## Step 1. 보안그룹 생성

EC2 콘솔 → 보안그룹 → 보안그룹 생성

```bash
이름: my-web-sg
VPC: my-vpc

인바운드 규칙
SSH      | TCP | 22   | 내 IP/32
HTTP     | TCP | 80   | 0.0.0.0/0
HTTPS    | TCP | 443  | 0.0.0.0/0
Spring   | TCP | 8080 | 0.0.0.0/0

아웃바운드 규칙
기본값 그대로 (모든 트래픽 허용)
```text
### 삽질 포인트 ①: 보안그룹이 안 보이는 경우

EC2 생성 화면에서 my-web-sg(보안그룹)가 목록에 안 뜨는 경우가 있다.
원인은 **VPC 설정** 때문이다.

```bash
보안그룹은 특정 VPC에 종속된다

my-web-sg → my-vpc 안에 묶여있음

EC2 생성할 때 VPC를 default로 선택하면
→ my-vpc 안에 있는 my-web-sg가 보이지 않음 ❌

VPC를 my-vpc로 선택해야
→ my-web-sg가 보임 ✅
```text
VPC, 서브넷, 보안그룹은 전부 세트로 묶여있다.
**EC2 만들 때 VPC부터 먼저 확인하는 습관**을 들여야 한다.

---

## Step 2. IAM 역할 생성

IAM 콘솔 → 역할 → 역할 생성

```bash
신뢰할 수 있는 엔터티 유형: AWS 서비스
사용 사례: EC2   ← 반드시 EC2 선택

권한 정책
→ AmazonS3FullAccess
→ AmazonRDSFullAccess

이름: my-ec2-role
```text
### 실무에서는 이렇게 안 한다

실습이라 FullAccess를 붙였지만 실무에서는 **최소 권한 원칙**을 따른다.

```bash
❌ 실습용
AmazonS3FullAccess → S3 모든 작업 가능

✅ 실무용
특정 버킷 읽기 권한만 → 딱 필요한 것만
```text
---

## Step 3. EC2 생성

EC2 콘솔 → 인스턴스 → 인스턴스 시작

```bash
이름: my-web-server
AMI: Amazon Linux 2023 (프리티어)
인스턴스 유형: t2.micro (프리티어)

키 페어: 새로 생성 (my-key.pem)

네트워크 설정
VPC: my-vpc
서브넷: public-subnet-a   ← 반드시 퍼블릭 서브넷
퍼블릭 IP 자동 할당: 활성화
보안그룹: my-web-sg

고급 세부 정보
IAM 인스턴스 프로파일: my-ec2-role
```text
### 키 페어 주의사항

```bash
.pem 파일은 딱 한 번만 다운로드 가능
잃어버리면 EC2 SSH 접속 영구 불가
깃허브에 절대 올리면 안 됨
```text
### 삽질 포인트 ②: SSH 접속이 안 되는 경우

EC2를 만들고 SSH 접속을 시도했는데 계속 연결이 안 됐다.
원인을 찾아보니 **서브넷을 프라이빗으로 설정**했기 때문이었다.

```bash
프라이빗 서브넷
→ 인터넷 게이트웨이 연결 없음
→ 퍼블릭 IP 자체가 없음
→ 외부에서 SSH 접속 불가 ❌

퍼블릭 서브넷
→ 인터넷 게이트웨이 연결 있음
→ 퍼블릭 IP 할당됨
→ 외부에서 SSH 접속 가능 ✅
```text
서브넷은 생성 후 변경이 불가능하다.
인스턴스를 종료하고 퍼블릭 서브넷으로 다시 만들어야 했다.

---

## Step 4. SSH 접속 확인

Git Bash에서 아래 명령어로 접속한다.

bash

```bash
# .pem 파일 권한 설정
chmod 400 ~/Downloads/my-key.pem

# SSH 접속
ssh -i ~/Downloads/my-key.pem ec2-user@<퍼블릭 IP>
```text
- .pem 파일권한을 읽기 권한만 주는 이유는 너무 많은 권한을 주게 되면 ssh에서 이 .pem 파일은 권한이 너무 많아서 위험하다고 거부를 한다.
    - 즉 사용자만 읽을 수 있는 권한을 주어 **다른 사람들이 .pem 파일 자체를 볼 수 없게 보안을 강화**해준다.
- 접속 성공하면 이런 화면이 뜬다.

## 오늘 삽질로 배운 것 정리

```bash
① 보안그룹은 VPC에 종속된다
  → EC2 만들 때 VPC 먼저 확인하는 습관 필요

② IAM 사용자 ≠ IAM 역할
  → 사람용 = 사용자 / 서비스용 = 역할

③ 프라이빗 서브넷은 퍼블릭 IP가 없다
  → SSH 접속하려면 반드시 퍼블릭 서브넷에 올려야 함
```text

## 마무리

오늘 만든 구조를 정리하면 이렇다.

```bash
my-vpc (10.0.0.0/16)
  └── public-subnet-a
        └── EC2 (my-web-server)
              ├── 보안그룹 (my-web-sg)
              └── IAM 역할 (my-ec2-role)
```text
다음 글에서는 이 EC2에 **RDS를 프라이빗 서브넷에 연결**하는 과정을 다룰 예정이다.
