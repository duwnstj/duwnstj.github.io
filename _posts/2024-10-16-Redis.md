## 컴퓨터의 구조 이해
Redis를 배우기 전에 **캐시**가 무엇인지 이해하는 것이 중요합니다. 
컴퓨터의 주요 구성 요소에는 다음과 같은 것들이 있습니다.

**CPU(중앙 처리 장치)** : 컴퓨터의 두뇌로 모든 연산을 처리해줍니다. 
**Memory(RAM)** : 임시 저장 장치로 CPU가 빠르게 접근해야할 데이터를 가지고 있습니다.
**Disk** : 영구 저장 장치로 데이터를 저장하는 역할을 합니다. 

![](https://velog.velcdn.com/images/duwnstj12/post/88e8ce8a-fb46-4dbe-94f8-1219f17aed0c/image.png)

- 위의 이미지에서 볼 수 있듯이 Redis는 메모리와 같은 임시 저장소 역할을 합니다. 

## 그럼 왜 Redis를 사용할까요?
메모리와 디스크의 IO 성능은 수십에서 수천배 차이가 납니다.

![](https://velog.velcdn.com/images/duwnstj12/post/c2da6ec4-3131-42a1-b4bb-c46d25cdb99e/image.png)

따라서 비록 메모리 데이터는 휘발 되지만 임시 저장소에서 데이터를 가져오는것이 훨씬 더 성능이 좋기 때문에 Redis가 사용됩니다.

### 캐시의 특징

#### 1. Disk에 비해서 성능이 훨씬 좋다
#### 2. Disk와 달리 값이 영구적으로 저장되지않는다.

- 캐시의 대표적인 서비스가 바로 **Redis**입니다.

## Redis
Redis는 Memory 기반의 Data 저장소입니다. 메모리에 데이터를 저장하기 때문에 **휘발성**을 가지고 있습니다.

### Redis의 특징
- **인메모리 저장소** : 데이터를 메모리에 저장해 매우 빠른 읽기/쓰기를 제공합니다.
- **다양한 데이터 구조 지원** : 문자열,해시,리스트,세트,정렬된 세트 등을 지원합니다.
- **데이터 지속성 옵션** : 메모리 데이터를 디스크에 저장할 수 있어 데이터 영구 저장이 가능합니다.
- **분산 시스템 지원** : 클러스터링과 복제 기능으로 확장성과 가용성을 높입니다.
- **PUb/Sub 및 Lua스크립팅** : 메시지 큐 ,실시간 통신에 유용한 Pub/Sub 기능과 서버측 스크립팅 기능을 제공합니다. 

- 이러한 다양한 데이터 구조 덕분에 개발자들은 다양한 요구 사항에 맞춰 데이터를 선택하여 사용할 수 있으며 이를 통해 비즈니스 로직을 간소화하고 유지 보수를 쉽게 할 수 있습니다. 

## Redis의 약점
Redis는 메모리 기반의 Data 저장소 이기 때문에 **데이터가 영구적으로 저장되지 않는다는 문제점**이 있습니다. 그럼 이 문제점을 Redis가 어떻게 해결했을까요? 

## 전략 1 : 특정 시점에 Disk(RDB)에 저장
이 방법은 Redis에 있는 데이터를 주기적으로 디스크에 저장하는 방식으로 **RDB 스냅샷**이라고도 합니다.

- **장점** :
    - 주기적으로 디스크에 저장하여 장애 발생시 복구 가능
    - 설정이 간단하고 관리가 쉬움

- **단점** : 
    - 스냅샷과 스냅샷 사이의 데이터는 손실될 수 있음
    - 데이터가 많을 경우 스냅샷 저장시 시스템 리소스 사용이 증가 할 수 있음 
    
## 전략 2 : 모든 로그를 기록(AOF)
- Redis의 모든 동작을 로그로 기록하고, 장애 발생시 로그를 사용하여 데이터를 복구하는 방법입니다.
    - 이런 방법을 `AOF(Append Only File)`라고 합니다.

- **장점** : 
    - 데이터 손실 가능성을 최소화
    - 장애 발생 시 최신 데이터를 복구 할 수 있음

- **단점** : 
    - 로그 파일 크기가 매우 커질 수 있음
    - 파일 크기가 커지면 복구 시간이 길어질 수 있고, 주기적으로 AOF 파일을 압축해야함
    
## 전략 3 : 데이터 복제(Master-Slave)
- Redis는 **Master-Slave** 구조를 통해 데이터를 분산 저장하여 데이터 손실을 방지 할 수 있습니다. 

![](https://velog.velcdn.com/images/duwnstj12/post/10295569-6385-4a84-a32b-f12b396fd2f2/image.png)

1개의 **Master Redis**에서 모든 요청을 처리하고 그 데이터를 **Slave Redis**에 복제합니다. 만약 Master Redis에 장애가 발생하면 Slave Redis가 Master로 승격되고 기존 Master는 복구 이후에 Slave로 강등 되어 계속 운영됩니다.

## Redis Sentinel
**Sentinel**은 Master-Slave 구조에서 Redis의 상태를 모니터링하고 장애 발생시 **failover**를 처리합니다.

![](https://velog.velcdn.com/images/duwnstj12/post/ee7f1f50-3b00-4188-8298-e83bb86ef065/image.png)

- Sentinel은 Master Redis의 상태를 주기적으로 확인하고 문제가 발생하면 자동으로 Slave Redis를 Master로 승격하여 장애를 처리합니다.

## Redis Cluster
**Redis Cluster**는 여러개의 Master-Slave 구조를 사용하여 데이터를 분산 저장하는 방식입니다. 이렇게 하면 더 많은 데이터를 안전하게 관리할 수 있습니다. 

![](https://velog.velcdn.com/images/duwnstj12/post/e58515da-e35c-42e4-8394-12b01cf93d5e/image.png)


## Redis가 사용되는 예시

### 뉴스 기사 캐싱
- 뉴스 데이터는 자주 조회되지만 자주 변경되지는 않기 때문에 **Redis**에 데이터를 저장하여 빠르게 조회 할 수 있습니다.

### 실시간 뉴스 업데이트
- Redis의 **Pub/Sub 기능**을 이용해 실시간 속보를 구독자에게 빠르게 전달 할 수 있습니다.
