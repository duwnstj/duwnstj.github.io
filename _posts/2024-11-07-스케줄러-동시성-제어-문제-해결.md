## 스케줄러 동시성 제어 문제 해결
![](https://velog.velcdn.com/images/duwnstj12/post/d22e6b96-8ffb-4079-9df3-95e804af8fd3/image.png)

### 동시성 제어가 필요한 이유
- 우리 프로젝트는 Docker 이미지를 GitHub Actions를 통해 Docker Hub에 저장하고, 이를 AWS EC2로 배포하는 CI/CD 자동 배포 환경을 구축하고 있습니다.
- 이렇게 인프라가 구축되어 있는 상황에서 서버 확장 시 기존 Docker 이미지를 활용해 새로운 서버를 만들 수 있습니다. 이때, 여러 서버가 동일한 스케줄러 작업을 동시에 실행할 위험이 있다고 판단하여 동시성 문제를 해결할 필요가 생겼습니다.

### 동시성 제어 문제 해결 방안
동시성 문제를 해결하기 위해 두가지 방법을 고려했습니다.

**1.API 서버와 스케줄러 서버를 분리**
- 스케줄러 서버를 별도로 분리하여 운영하는 방법을 우선 고려했습니다.
- 그러나 우리 서비스에서 스케줄러로 처리해야할 데이터 양이 많지 않다고 판단하여 비용과 효율성을 고려했을 때, 서버를 두개로 나누는 것은 적합하지 않다고 판단했습니다.

**2. Redisson 분산 락(Distributed Lock) 사용**
- 분산 락은 지정한 시간 동안 특정 리소스에 대한 락을 걸어 다른 스레드가 해당 리소스에 접근할 수 없도록 막아주는 방식입니다. 이를 통해 스케줄러의 동시 실행 문제를 효과적으로 해결 할 수 있겠다고 판단했습니다.
- Redisson은 Redis를 사용하여 락 상태를 관리하므로 분산 시스템 환경에서도 락을 유지할 수 있다는 장점이 있습니다.

## Redisson을 사용하기 위한 환경설정
- Redisson을 사용하려면 먼저 build.gradle에 의존성을 주입해주어야합니다.

  ```java
  //redisson
    implementation 'org.redisson:redisson-spring-boot-starter:3.17.6'
```text
    
## RedissonConfig 설정
Redis와의 연결을 위한 설정 클래스입니다.RedissonClient를 빈으로 등록하여 다른 클래스에서 사용할 수 있도록 합니다.
```java
package com.example.library_management.global.config;

import org.redisson.Redisson;
import org.redisson.api.RedissonClient;
import org.redisson.config.Config;
import org.springframework.beans.factory.annotation.Value;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

@Configuration
public class RedissonConfig {

    @Value("${spring.data.redis.host}")
    private String redisHost;

    @Value("${spring.data.redis.port}")
    private int redisPort;

    @Value("${spring.data.redis.password}")
    private String password;

    @Bean
    public RedissonClient redissonClient() {

        //redisson 설정을 정의하는데 사용
        Config config = new Config();

        // Redis 연결 설정
        config.useSingleServer()
                .setAddress("redis://" + redisHost + ":" + redisPort)
                .setPassword(password)
                .setConnectionMinimumIdleSize(1)
                .setConnectionPoolSize(2)
                .setRetryAttempts(3)
                .setRetryInterval(1500)
                .setDnsMonitoringInterval(5000)
                .setSubscriptionConnectionMinimumIdleSize(1)
                .setSubscriptionConnectionPoolSize(2)
                .setTimeout(3000);

        return Redisson.create(config);

    }
}

```text
- 이 `RedissonConfig` 클래스는 Redis 서버와 연결을 설정하고 RedissonClient를 빈으로 등록하여 스케줄러에서 사용할 수 있도록 해줍니다.

## 스케줄러 코드 구현
아래는 Redisson 분산 락을 적용하여 구현한 스케줄러 코드입니다.
```java
@Service
@RequiredArgsConstructor
@Slf4j
public class NotificationScheduler {

    private final RentalNotifyService rentalNotifyService;
    private final StudyRoomNotifyService studyRoomNotifyService;
    private final RedissonClient redissonClient;

    //매일 오전 8시 30에  대여 및 스터디룸 예약 알림을 자동으로 전송
    @Scheduled(cron = "0 30 8 * * ?")
    public void sendReminders() {


        RLock lock = redissonClient.getLock("NotificationSchedulerLock");
        try {
            //watchDog로 락 갱신 연장
            lock.lock(10, TimeUnit.MINUTES);
            //대여 알림 전송
            rentalNotifyService.sendRentalReminders();
            //스터디룸 예약 알림 전송
            studyRoomNotifyService.sendReservationReminders();
        } catch (Exception e) {
            log.error("알림 전송 중 에러가 발생했습니다. ", e);
        } finally {
            if (lock.isHeldByCurrentThread()) {// 현재 스레드가 락을 보유하고 있는 지 확인
                lock.unlock(); // 락을 보유하고 있다면 락 해제
            }

        }
    }
```text
`lock()` 메서드를 사용한 이유
- 락을 걸어주는 방식에는 `tryLock()`과 lock()` 두가지가 있습니다.

- `tryLock()`의 경우 설정한 시간이 지나면 작업이 완료되지 않아도 락이 해제되는 문제가 있습니다. 반면 `lock()`은 작업이 끝날 때까지 락을 유지하며 Watchdog 기능을 통해 필요시 자동으로 락 시간을 연장할 수 있습니다.
- 이 방식으로 설정된 락은 다른 스레드가 접근할 경우 무한 대기 상태가 되어 여러 인스턴스에서 동시에 스케줄러 작업이 실행되는것을 방지할 수 있습니다.

## 정리 및 향후 개선 방향
- **정리** : Redisson을 사용한 분산 락은 코드 변경만으로 동시성 문제를 효과적으로 해결할 수 있는 좋은 방법이었습니다.
- **향후 개선 방향**: 서비스 규모가 커지거나 스케줄러 작업량이 증가할 경우, 다음과 같은 방안을 고려해볼 수 있습니다:

- **API 서버와 스케줄러 서버 분리**: 스케줄러 전용 서버를 별도로 분리하여, 스케줄러 로직의 부하를 줄이고 API 서버의 처리 성능을 높일 수 있습니다.

**서버리스 아키텍처**: AWS Lambda와 같은 서버리스 서비스를 활용하여 스케줄링 작업을 처리할 수도 있습니다. 서버리스는 필요에 따라 자동으로 확장할 수 있어 서비스 규모에 맞는 유연한 리소스 관리가 가능합니다.
