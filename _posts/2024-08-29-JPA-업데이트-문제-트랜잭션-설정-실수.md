개발을 하다 보면 예상치 못한 문제가 발생할 때가 많다.
오늘은 JPA를 사용해 데이터를 수정하려 했으나, 트랜잭션 설정 실수로 인해서 업데이트가 되지 않는 문제를 겪었다. 
이 글에서는 문제의 원인과 해결방법, 그리고 **트랜잭션을 서비스 계층에서 설정해야하는 이유**에 대해 설명해보겠다.

## 문제 발생 상황

### Postman 테스트 중 발견된 문제
프로젝트에서 일정 저장과 수정 기능을 구현했다. 저장 기능과 수정 기능 모두 트랜잭션을 사용해 데이터의 일관성을 유지하려고 했다. Postman을 이용해 API 테스트를 진행한 결과 , 저장은 잘 되었으나 **수정 작업에서는 데이터베이스에 전혀 반영되지 않는 문제**가 발생했다.
![](https://velog.velcdn.com/images/duwnstj12/post/ff8d7ba8-9019-461b-9245-861a813d24c5/image.png)

처음에는 데이터가 잘못 전달되었거나 코드에 문제가 있을 것이라 생각 했고, 디버깅을 해봤지만 requestDto 객체로 값이 잘 넘어오고 있었고 , 결국 원인은 트랜잭션을 잘못 설정한데에 있었다.

## 문제의 원인 : 잘못된 트랜잭션 설정
### 트랜잭션 설정의 실수
문제는 트랜잭션을 컨트롤러와 서비스 계층에 각각 나누어 적용한 데에서 발생했다.

1. **저장 로직**: 컨트롤러에서 데이터를 저장하는 메소드에 `@Transactional` 애노테이션을 적용했다.
2. **수정 로직**: 서비스 계층에서 데이터를 수정하는 메서드에 `@Transactional` 애노테이션을 적용했다.

이렇게 트랜잭션을 나누어 적용하면, 트랜잭션이 제대로 전파되지 않거나, 트랜잭션의 범위가 애매해져 예상한대로 동작하지 않을 수 있다. 이로 인해 수정 작업에서 트랜잭션이 올바르게 작동하지 않아, 데이터 베이스에 변경 사항이 반영되지 않았던 것이였다.

### 트랜잭션의 역할
트랜잭션은 데이터베이스에서 여러 작업을 하나의 단위로 묶어주는 기능이다. 트랜잭션이 잘못 설정되면, 이러한 묶음이 깨져서 일부 작업이 반영되지 않거나 데이터 일관성이 깨질 수 있다.

## 개선전 코드
### 기존 ScheduleController 코드

{% highlight java %}
@RestController
@Transactional(readOnly = true) // 전체 컨트롤러에 트랜잭션이 적용됨
@RequestMapping("/api/schedules")
public class ScheduleController {

    private final ScheduleService scheduleService;

    public ScheduleController(ScheduleService scheduleService) {
        this.scheduleService = scheduleService;
    }

    @Transactional // 저장 로직에 트랜잭션 적용
    @PostMapping
    public ResponseEntity<ScheduleSaveResponseDto> createSchedule(@RequestBody ScheduleSaveRequestDto scheduleRequestDto) {
        ScheduleSaveResponseDto savedSchedule = scheduleService.saveSchedule(scheduleRequestDto);
        URI location = URI.create("/api/schedules/" + savedSchedule.getId());
        return ResponseEntity.created(location).body(savedSchedule);
    }

    @PutMapping("/{id}")
    public ResponseEntity<ScheduleUpdateResponseDto> upgradeSchedule(
            @PathVariable Long id, @RequestBody ScheduleUpdateRequestDto updateRequestDto) {
        ScheduleUpdateResponseDto upgradeSchedule = scheduleService.upgradeSchedule(id, updateRequestDto);
        return ResponseEntity.ok(upgradeSchedule);
    }
}

{% endhighlight %}
## 기존 ScheduleService 코드

{% highlight java %}
@Service
public class ScheduleService {

    private final ScheduleRepository scheduleRepository;

    public ScheduleService(ScheduleRepository scheduleRepository) {
        this.scheduleRepository = scheduleRepository;
    }

    @Transactional // 수정 로직에 트랜잭션 적용
    public ScheduleUpdateResponseDto upgradeSchedule(Long id, ScheduleUpdateRequestDto updateRequestDto) {
        Schedule getSchedule = scheduleRepository.findById(id).orElseThrow(() ->
                new NullPointerException("아이디를 찾을 수 없습니다."));
        getSchedule.update(
                updateRequestDto.getContent(),
                updateRequestDto.getTitle(),
                updateRequestDto.getUserName()
        );
        return new ScheduleUpdateResponseDto(getSchedule.getId());
    }
}

{% endhighlight %}
위 코드에서는 트랜잭션이 저장 로직에서는 컨트롤러에, 수정 로직에서는 서비스에 적용되어 있었다. 이런 설정으로 인해 수정 로직이 정상적으로 동작하지 않았던 것이였다.

## 트랜잭션을 서비스 계층에서 설정해야하는 이유
### 1. 비즈니스 로직의 일관성 유지
트랜잭션은 데이터베이스 작업을 하나의 단위로 묶어주므로 , 비즈니스 로직을 처리하는 서비스 계층에서 관리하는 것이 적절하다. 이렇게 하면 여러 데이터베이스 작업이 일관성 있게 처리 될 수 있다.

### 2. 역할의 명확한 분리
컨트롤러는 사용자의 요청을 처리하는 역할에 집중하고, 서비스는 비즈니스 로직과 데이터베이스 작업의 일관성을 관리한다. 이처럼 역할을 명확하게 분리하면 코드의 유지보수가 쉬워지고, 예상치 못한 오류를 방지할 수 있다.
### 3. 트랜잭션 전파 문제 방지
트랜잭션은 전파(Propagation)설정에 따라 다른 계층으로 전파될 수 있다. 서비스 계층에서 트랜잭션을 관리하면, 컨트롤러와 서비스 사이에서 트랜잭션이 제대로 전파되지 않는 문제를 방지 할 수 있다.

## 해결 방법 : 트랜잭션을 서비스 계층에서 일원화
트랜잭션 관리를 컨트롤러에서 제거하고 ,모든 트랜잭션 로직을 서비스 계층에서 처리하도록 변경했다. 이렇게 하면 트랜잭션의 전파 문제를 해결하고, 데이터 베이스 작업의 일관성을 유지할 수 있다.

### 개선된 ScheduleService 코드
{% highlight java %}
@Service
public class ScheduleService {

    private final ScheduleRepository scheduleRepository;

    public ScheduleService(ScheduleRepository scheduleRepository) {
        this.scheduleRepository = scheduleRepository;
    }

    @Transactional // 트랜잭션을 서비스 계층에서 일원화
    public ScheduleSaveResponseDto saveSchedule(ScheduleSaveRequestDto scheduleRequestDto) {
        Schedule schedule = new Schedule(scheduleRequestDto);
        Schedule savedSchedule = scheduleRepository.save(schedule);
        return new ScheduleSaveResponseDto(savedSchedule);
    }

    @Transactional // 수정 로직에도 트랜잭션 적용
    public ScheduleUpdateResponseDto upgradeSchedule(Long id, ScheduleUpdateRequestDto updateRequestDto) {
        Schedule getSchedule = scheduleRepository.findById(id).orElseThrow(() ->
                new NullPointerException("아이디를 찾을 수 없습니다."));
        getSchedule.update(
                updateRequestDto.getContent(),
                updateRequestDto.getTitle(),
                updateRequestDto.getUserName()
        );
        return new ScheduleUpdateResponseDto(getSchedule.getId());
    }
}

{% endhighlight %}
## 해결 결과 
코드를 개선한 후 , Postman을 이용해 다시 테스트 해본 결과 , 수정 로직이 정상적으로 동작하고 데이터베이스에도 올바르게 반영되었다. 트랜잭션을 서비스 계층에서 관리하면서 코드의 구조가 더 깔끔해졌고, 트랜잭션의 범위가 명확해져 문제가 해결되었다.

## 결론
이번 트러블 슈팅을 통해 트랜잭션을 어디에 적용해야하는지에 대한 중요성을 다시 한번 깨달았다. 트랜잭션은 데이터의 일관성을 유지하는데 매우 중요한 역할을 한다. 이 기능을 제대로 활용하기 위해서는 비즈니스 로직을 담당하는 서비스 계층에서 트랜잭션을 관리하는 것이 좋다. 앞으로는 이러한 경험을 바탕으로 더 안정적이고 확실한 코드를 작성할 수 있을 것 같다.
