 최근 지인의 부탁으로 **Claude Code**를 활용해 제작된 웹 서비스를 점검할 기회가 있었습니다. "AI가 만들었으니 코드 자체는 깔끔하겠지"라는 생각으로 접근했지만, 인프라 아키텍트를 꿈꾸는 제 시선에 비친 실상은 생각보다 훨씬 위태로웠습니다.

이번 작업은 단순히 '코드 몇 줄'을 고치는 과정이 아니었습니다. **AI라는 도구는 '답'을 주지만, 그 답이 '안전한 아키텍처' 위에서 돌아가게 만드는 것은 결국 아키텍트의 통찰력**이라는 점을 증명하는 시간이었습니다.

---

## 1. 아키텍처 진단: 겉은 멀쩡해도 속은 비어있던 설계

가장 먼저 제 '아키텍트 레이더'가 요동친 건 불안정한 트래픽 경로였습니다. 네이버에선 괜찮은데, 유독 구글에서만 **옛날 도메인 주소**로 유입되고 있었죠. 클릭하면 보안이 취약한 과거 주소로 연결되는 기묘한 상황이 벌어지고 있었습니다.

### 🚩 트래픽 경로의 파편화 (Traffic Path)

- **현상:** 구 도메인 유입 지속 및 HTTPS 강제 접속 시 **406 Error** 발생.
- **원인:** 신구 도메인 간 **301 포워딩 정책** 부재 및 **HTTP(80) → HTTPS(443)** 리다이렉션 설정 누락.
- **아키텍트의 분석:** "구글 인덱싱 탓이 아니야. 도메인이 변경되었다면 구 주소 트래픽을 현 주소로 강제 연결하는 설정이 서버 단에서 완벽했어야 해. 이건 인프라 설계의 결함이야."

### 🚩 보안의 착각 (Encryption vs Authentication)

DB 내부를 확인했을 때 데이터들은 잘 암호화되어 있었습니다. 친구는 "암호화됐으니 안전하다"며 안심했지만, 저는 더 큰 구멍을 보았습니다.

- **현상:** 저장 시 암호화(At-Rest)는 되어 있으나, 요청 주체를 검증할 **Security/Refresh Token** 로직 전무.
- **아키텍트의 분석:** 자물쇠는 최고급인데 열쇠는 문틈에 끼워둔 꼴입니다. 인증 체계가 없는 암호화는 세션 탈취 한 번에 모든 권한을 넘겨줄 수 있는 '반쪽짜리' 보안입니다.

## 2. 운영의 민낯: "GitHub은 장식, 본체는 FTP였네?"

아키텍트로서 가장 경악했던 지점은 서비스의 **지속 가능성(Sustainability)**을 담보하는 배포 프로세스였습니다.

- **배포 파이프라인 부재:** GitHub 코드가 최종본인 줄 알았으나, 실제로는 **FTP로 서버 파일을 직접 수정**하는 파편화된 방식으로 운영되고 있었습니다. 형상 관리가 없는 시스템은 언제 터질지 모르는 시한폭탄과 같습니다.
- **리소스 구조의 무질서:** 로고 이미지 같은 정적 자산이 폴더 구분도 없이 소스 코드와 뒤섞여 루트 디렉토리에 흩어져 있었습니다. 향후 확장성(Scalability)을 고려해 직접 이미지 전용 폴더를 설계하고 자산 경로를 재배치하는 기초 공사부터 다시 시작해야 했습니다.

---

## 3. Solution: 아키텍트가 고집한 '안전한 개선' 원칙

운영 중인 시스템의 무결성을 지키기 위해 다음과 같은 엄격한 5단계 정상화 프로세스를 밟았습니다.

1. **Isolation (환경 격리):** `main` 브랜치를 직접 건드리지 않고 별도의 **보안 개선 전용 브랜치** 생성. 운영 리스크 최소화는 아키텍트의 기본 소양입니다.
2. **Structuring (구조 설계):** 유지보수를 위해 이미지 전용 폴더 설계 및 자산 경로 재배치. 시스템 레이아웃의 표준을 잡는 과정이었습니다.
3. **AI Orchestration (리딩):** 제가 정의한 보안 취약점 범주를 기반으로 Claude Code가 코드를 생성하도록 가이드. AI의 속도를 내 지식으로 컨트롤했습니다.
4. **Verification (전수 검증):** 보안 패치가 기존 비즈니스 로직에 영향을 주지 않는지 로컬에서 **모든 API 기능 테스트** 수행.
5. **Synchronization (형상 일치):** 검증 완료 후 FTP 배포 및 GitHub `main` Merge를 통해 최종적으로 소스 코드 형상을 일치시키고 작업 브랜치 삭제.

## 4. 회고: 클라우드 아키텍트는 무엇으로 증명하는가?

이번 작업에서 Claude Code의 퍼포먼스는 압도적이었습니다. 하지만 동시에 깊은 고민이 생겼습니다. **"이제 단순히 코딩 언어를 공부하거나 사용법만 아는 수준이라면 곧바로 대체되겠구나."**

친구도 똑같은 Claude Code를 써서 사이트를 만들었지만 이런 구멍들을 잡아내지 못했습니다. 차이는 **'의문'을 품을 수 있는 지식의 유무**였습니다. 인프라적 통찰이 없었다면 다음과 같은 질문 자체를 할 수 없었을 테니까요.

- *"왜 80포트 리다이렉트가 안 돼?"*
- *"암호화는 되어 있는데 검증 로직은 어디 있어?"*

## 5. 마치며: 'How'가 아닌 'What & Why'를 증명하는 사람

모든 보안 문제를 해결하고, 도메인 포워딩의 필요성을 인지시키며 작업을 마쳤습니다. 마지막에 작업 브랜치를 Merge 하고 삭제하며 느낀 쾌감은 단순한 코드 수정 그 이상이었습니다.

AI 시대의 아키텍트는 구현자가 아니라, **구현된 결과물이 올바른 설계 안에서 안전하게 움직이는지 증명하는 사람**이어야 합니다. 이제는 'How to code'를 넘어 **'What to check & Why it matters'**를 고민하는 아키텍트가 되어야겠습니다.
