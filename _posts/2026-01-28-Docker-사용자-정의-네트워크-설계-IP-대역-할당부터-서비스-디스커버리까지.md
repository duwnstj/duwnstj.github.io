클라우드 아키텍트를 목표로 리눅스의 구조적 원리를 공부하다 보면, 도커(Docker)라는 도구가 단순히 편리한 컨테이너 엔진을 넘어 **리눅스 네트워크 체계의 집약체**라는 사실을 깨닫게 됩니다. 이번 포스팅에서는 도커 네트워크 생성부터 멀티 네트워크 연결까지, 그 이면에서 동작하는 리눅스 커널의 역할을 정리해 보겠습니다.

---

## 1. 네트워크 설계와 컨테이너 배치 (Deployment)

도커에서 사용자 정의 네트워크를 만드는 것은 인프라의 '세계관'을 설계하는 작업입니다.

```text
# 특정 IP 대역을 지정한 'vswitch-net' 생성
$ docker network create \
  --driver bridge \
  --subnet 172.30.1.0/24 \
  --ip-range 172.30.1.0/24 \
  --gateway 172.30.1.1 \
  vswitch-net
```
- **`-subnet`**: **"우리 동네 전체 크기"**를 정하는 설계도입니다. `/24`는 256개의 주소 공간을 확보함을 의미합니다.
- **IP 할당 전략**:
    - **자동 할당**: `docker run --net=vswitch-net ...` (도커가 남는 번호를 임시 부여)
    - **직접 명시**: `docker run --net=vswitch-net --ip 172.30.1.100 ...` (운영자가 고정 주소 부여)

---

## 2. 서브넷 마스크(/26)의 본질: '개수'가 아닌 '구역'

많은 입문자가 `/26` 마스크를 "IP를 62개만 쓸 수 있는 제약"으로 오해하곤 합니다. 하지만 아키텍트의 관점에서 서브넷 마스크는 **논리적인 경계선**입니다.

- **100번 IP의 소속**: 네트워크를 `/26`으로 쪼개면 64개씩 그룹화됩니다. 이때 `172.30.1.100` 주소는 수량 제한에 걸리는 것이 아니라, **2번 구역(64~127번 동네)**의 주민으로 할당되는 것입니다.
- **통신 격리**: 설계도를 `/24`에서 `/26`으로 바꾸는 순간, 이웃이었던 1번 주소는 '다른 동네'가 되어 직접 통신이 차단됩니다. 이것이 인프라 레벨 보안의 기초입니다.

---

## 3. 구조적 이해: docker0와 사용자 정의 네트워크의 병렬성

많은 이들이 `docker0`를 모든 도커 통신의 관문으로 생각하지만, 사실 **사용자 정의 네트워크는 docker0와 병렬적으로 존재하는 별도의 가상 스위치**입니다.

> ![](https://velog.velcdn.com/images/duwnstj12/post/3c948205-9117-4f17-9de3-860f93c4e769/image.png)
> 

위의 `brctl show` 결과를 보면 알 수 있듯이, 리눅스 호스트 입장에서는 `docker0`와 `br-5abfe...`(vswitch-net)라는 두 장치가 독립적으로 존재합니다. 데이터는 목적지 IP에 따라 리눅스 커널로부터 직접 해당 스위치로 전달되며, 내가 만든 네트워크는 `docker0`를 거치지 않습니다.

---

## 4. 데이터의 여정: 도커 뒤에 숨은 리눅스 커널

도커 네트워크는 리눅스 커널의 기능을 빌려 쓰는 '세입자'에 불과합니다. 전체 아키텍처를 시각화하면 다음과 같습니다.

> ![](https://velog.velcdn.com/images/duwnstj12/post/bfb0fc40-4116-4cf9-a2cf-2cd17757e185/image.png)
>

1. **가상머신 입구**: `enp0s8` 인터페이스로 데이터 도착
2. **교통 정리**: 리눅스 커널의 **`iptables`*가 데이터를 가로채서 경로 확인 (NAT/Forwarding)
3. **브리지 통과**: 목적지 IP를 확인하고 `docker0`를 건너뛰어 곧장 **`vswitch-net`*으로 전달
4. **최종 도달**: 도커 DNS/Proxy의 안내를 받아 실제 컨테이너 내부로 진입
- 결국 **도커 네트워크 체계는 리눅스 네트워크 체계를 그대로 따릅니다.** 아키텍트가 리눅스 커널을 깊이 이해해야 하는 이유가 바로 여기에 있습니다.
---

## 5. 멀티 네트워크 연결: 한 컨테이너에 두 개의 심장(NIC) 달기

실무에서는 보안상의 이유로 웹 서버가 외부망과 내부 DB망에 동시에 연결되어야 할 때가 있습니다. 이때 `docker network connect`를 사용합니다.

> ![](https://velog.velcdn.com/images/duwnstj12/post/30c25c75-cfac-48c2-a3ff-8e74b33110d7/image.png)
>

```text
# 실행 중인 컨테이너에 새로운 네트워크 대역 추가 연결
$ docker network connect fc-net2 add-net
```
- **결과**: `ifconfig`로 확인해 보면 컨테이너 내부에 `eth0` 외에 **`eth1`**이 새롭게 생성되어 두 개의 IP를 가지게 됩니다.
- **의미**: 하나의 컨테이너가 서로 다른 두 스위치에 동시에 연결되어, 격리된 네트워크 사이의 통로 역할을 할 수 있습니다.

---

## 6. 핵심: Docker DNS를 통한 서비스 디스커버리

사용자 정의 네트워크의 최대 장점은 IP가 아닌 **컨테이너 이름으로 통신이 가능하다**는 점입니다.

> ![](https://velog.velcdn.com/images/duwnstj12/post/aefe67ed-9b94-4565-9191-6080a3685bda/image.png)
> 

실습 결과, `net-check1`에서 `net-check2`라는 이름으로 핑을 날리면 도커 DNS가 즉시 IP(`172.18.0.3`)를 찾아 연결해 줍니다. IP는 유동적이지만 이름은 고정되므로 서비스의 안정성이 보장됩니다.

---

## 7. [Validation] 운영자로서의 핵심 체크포인트

- **인터페이스 검증**: `brctl show`를 통해 각 브리지에 어떤 컨테이너(`veth`)가 연결되었는지 상시 모니터링할 것.
- **이중 연결 주의**: 한 컨테이너가 여러 네트워크에 연결될 경우 라우팅 테이블이 복잡해지므로 우선순위를 체크할 것.
- **커널 레벨 확인**: 통신 장애 시 컨테이너 내부가 아닌 호스트의 `iptables` 규칙을 먼저 점검할 것.
