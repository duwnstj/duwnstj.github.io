- **N+1 문제**는 JPA에서 **Lazy Loading**으로 인해 발생하는 성능 문제중 하나입니다. 기본적으로 하나의 엔티티를 조회한 후 , 연관된 엔티티 N개를 각각 개별적으로 추가 쿼리로 로드할 때 발생하는 문제입니다. 예를 들어 부모 엔티티를 한번 조회하고 각 부모 인테티와 연관된 자식 엔티티들을 추가로 로딩할 때 총 N개의 추가 쿼리가 발생하게 됩니다. 

## 해결 방법

### 1. Fetch Join 사용
- **Fetch Join**을 사용하면 한번의 쿼리로 연관된 엔티티들을 함께 가져올 수 있습니다. `@Query`를 사용하여 JPQL로 `join fetch`를 명시하면 N+1문제를 해결 할 수 있습니다.

- 예시 : 

{% highlight java %}
@Query("SELECT p FROM Parent p JOIN FETCH p.children WHERE p.id = :id")
Parent findByIdWithChildren(@Param("id") Long id);

{% endhighlight %}
- **장점** : 추가적인 쿼리 발생을 방지하고 , 성능을 크게 향상시킬 수 있습니다.
- **단점** : Fetch Join은 가져오는 데이터의 양이 많아질 경우 성능 저하나 메모리 과부하가 발생할 수 있으므로 연관 데이터가 적절하게 필터링 되었는지 확인해야합니다.

### 2. EntityGraph 사용
- `@EntityGraph`를 통해 JPA가 연관된 엔티티를 함께 로딩하도록 지정할 수 있습니다. 이를 사용하면 간결한 코드로 **Eager Fetch**를 구현할 수 있으며 특정 엔티티의 연관 필드를 로딩하는 방법을 제어할 수 있습니다.
- **예시** : 

{% highlight java %}
@EntityGraph(attributePaths = {"children"})
@Query("SELECT p FROM Parent p WHERE p.id = :id")
Parent findByIdWithChildren(@Param("id") Long id);

{% endhighlight %}
- **장점** : 코드의 가독성이 높아지고, 특정 상황에서 Fetch 전략을 쉽게 제어할 수 있습니다.
- **단점** : 복잡한 쿼리에서는 한계가 있을 수 있으며, 대규모 데이터에서는 성능 문제가 발생할 수 있습니다.

### 3. Batch Size 설정
- `@BatchSize` 어노테이션 또는 `hibernate.default_batch_fetch_size` 설정을 통해 한번에 로드할 연관 엔티티의 수를 제한할 수 있습니다. 이를 통해 Lazy Loading 시 한번에 여러개의 연관 엔티티를 로드해 N+1 문제를 완화할 수 있습니다

- **예시** :

{% highlight java %}
@BatchSize(size = 10)
private List<Child> children;

{% endhighlight %}
- **장점** : 다수의 연관 엔티티를 한번에 가져와 성능을 개선할 수 있습니다.
- **단점** : 너무 큰 batch size 설정은 성능을 저하시킬 수 있으므로 적절한 크기로 조정해야합니다. 

## 꼬리 질문 예상

### 1. Fetch Join을 사용할 때 주의 할 점은 무엇인가요?
- Fetch Join을 사용할 떄는 연관된 데이터의 양을 고려해야합니다. 너무 많은 데이터를 한번에 가져오면 메모리 과부하가 발생할 수 있고 , 성능이 저하될 수 있습니다. 또한, 한 쿼리에 최대 하나의 컬렉션만 Fetch Join할 수 있으며 그렇지 않으면 Cartesian Product 문제가 발생할 수 있습니다.

### 2. EntityGraph와 Fetch Join의 차이점은 무엇인가요?

- **Fetch Join**은 JPQL을 사용하여 쿼리 내에서 명시적으로 연관된 엔티티를 가져오는 방법이고, **EntityGraph**는 JPA 어노테이션을 통해 로딩 전략을 제어하는 방식입니다. EntityGraph는 JPQL에 의존하지 않고 엔티티의 필드를 직접 지정해 로딩할 수 있어 코드 가독성이 좋지만 복잡한 조건을 처리하는데 Fetch Join만큼 유연하지는 않을 수 있습니다.

### 3. N+1 문제를 사전에 방지하려면 어떤 전략을 사용할 수 있나요?
- Fetch 전략을 미리 고려하여 설계하는 것이 중요합니다. 기본적으로 Lazy Loading을 사용하고 필요한 경우 Fetch Join이나 EntityGraph 등을 활용해 N+1 문제를 미리 방지 할 수 있습니다. 또한 , `@BatchSize`나 `hibernate.default_batch_fetch_size` 같은 설정을 통해 연관 데이터를 효율적으로 가져올 수 있습니다.

### 4. Batch Size 설정의 적정크기는 어떻게 결정하나요?

-Batch Size는 데이터베이스 성능과 메모리 사용량을 고려하여 설정해야합니다. 너무 작으면 N+1 문제 해결에 충분하지 않고 ,너무 크면 메모리 과부하를 유발할 수 있습니다. 보통 10~100 정도의 값을 권장하지만 , 프로젝트 상황에 맞춰 조정해야합니다.
