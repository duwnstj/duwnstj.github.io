## 병합을 위해 사용한 방법
- **PR(Pull Request)** : 
 우리 조는 Git 기능들을 많이 활용해보기 위해서 `Dev 브랜치`에는 무조건 PR(Pull Request)를 통해 병합을 해야하는 규칙을 Github의 rules라는 기능을 통해 정했다. 
- PR을 통해 병합을 진행하니 **팀원들과 코드 리뷰하기에도 좋았고 rules 기능을 사용하면 rules를 통해 승인할 수 있는 인원을 정해 병합 승인을 받을 수 있다는 점이 좋았다.** 

<hr>

## 팀프로젝트를 하며 생긴 문제점
1. **스마트 병합의 문제** :
    - IntelliJ의 `스마트 병합` 기능을 사용하여 병합을 했으나 점수 로직이 사라지는 문제가 발생했다. 스마트 병합기능이란? **IntelliJ**가 자동으로 충돌을 해결하려는 기능으로 이를 잘 이해하지 못한 상태에서 사용했던것이 문제였다.
2. ** 푸시되지 않은 코드를 병합한 문제** :
    - GitHub에 푸시하지 않은 상태에서 병합을 진행하여 , 병합이 완료된 후 코드가 사라지는 문제가 발생했다. 그 이유는 푸시가 되지 않은 로컬의 변경 사항이 병합을 하면서 손실 되었기 때문이다.

## 문제 해결 방법
1. **스마트 병합 문제 해결** : 
    - 인텔리제이의 `스마트 병합`을 뭔지 자세히 모르는 상태에서 스마트 병합을 해버리니 점수 수정을 위한 로직 코드가 전부 사라지는 경우가 발생했다. 
    - 혹시 몰라 백업용으로 코드를 복사해둬서 다행이였다. 이 `스마트 병합`이라는 것은 인텔리제이가 알아서 판단해서 병합 충돌이 나지 않게 수정을 하는 방법이였다.. 
    - 백업용으로 복사한 코드를 이용하여 다시 푸쉬 후 병합 충돌을 해결하고 병합을 해서 문제를 해결했다.
    - 나중에 튜터님께 여쭤보니 **스마트 병합은 푸쉬가 안된 상태이거나 병합 진행이 잘못되었을 때 나타나는 메세지라며 이 기능을 이용하지 않는 게 좋을 것같다고 피드백을 해주셨다.**
    <hr>
    
2. **푸시되지 않은 코드 복구** : 
- Dev 브랜치로 병합을 하기 전 기능 구현한 브랜치를 푸시를 하지 않고 바로 병합을 해버리니 이 때도 `스마트 병합`이라는 메세지가 떳다. 
- 이번엔 스마트 병합을 하지 않고 다시 병합을 했는데 Dev 브랜치를 확인해보니 코드가 사라진 문제점이 있었다. 
- 기능을 구현한 코드가 푸쉬가 되지 않았기 때문에 기능을 구현하기 전 코드를 Dev에 병합을 하는 상황이 나오게 된것이다. 
- 이번 방법은 튜터님께서 알려주신 인텔리제이의 `로컬기록`이라는 기능을 통해 복구 하였다.

## 인텔리제이 로컬기록 사용법
1. **로컬 기록 보기** : 
    - 인텔리제이에서 우클릭하여 나타나는 메뉴에서 **로컬기록**을 선택한다.
![](https://velog.velcdn.com/images/duwnstj12/post/a1001e66-69c7-4959-97ec-30a178b39a4a/image.png)
2. **복구하기** : 
    - 여기서 로컬기록의 기록 표시를 클릭하면 이전에 작성했던 코드들을 확인하고 , 필요한 부분을 복구할 수 있다.<br><br>
    
3. **Github에 푸시를 못했을 때** :
    - 만약 푸쉬를 했더라도 로컬 기록은 로컬에서 작성한 기록들을 가져오는것이기 때문에 코드 복구가 가능하다.

## 느낀점 
- **푸시 확인 습관** :기능 구현을 완료한 후 , 먼저 브랜치를 푸시한 후 병합을 진행해야한다는 점을 깨달았다. 푸시되지 않은 상태에서 병합하면 코드가 손실 될 수 있다.
- **로컬 기록 활용**: IntelliJ의 로컬 기록 기능을 통해 코드 복구가 가능하지만, 먼저 푸시를 확인하고 병합하는 습관을 들이는 것이 더 중요하다.
- **스마트 병합 사용 주의**: IntelliJ의 스마트 병합 기능은 신중하게 사용해야 하며, 일반적으로는 수동으로 충돌을 해결하는 것이 좋다.
