## 서론: 통합 과정의 문제와 CI의 필요성

소프트웨어 개발은 여러 개발자가 협업하여 코드를 작성하고 통합하는 작업입니다. 과거에는 각자가 작성한 코드를 합치는 과정에서 수많은 오류가 발생했습니다. 이러한 통합 오류는 문제점을 찾고 해결하는 데 많은 시간과 반복 작업을 요구했습니다.

이러한 비효율성을 개선하기 위해 **CI(Continuous Integration), 즉 지속적 통합**이라는 개념을 도입했습니다.

CI(Continuous Integration)란 무엇인가?

CI는 개발자들이 작성한 코드들을 중앙 저장소에 통합할 때, 발생할 수 있는 문제점들을 자동으로 감지하여 개발자들에게 알려주는 시스템입니다. 이를 통해 개발자는 어느 부분에서 오류가 발생했는지 빠르게 파악하고, 문제 해결에 소요되는 시간을 현저히 줄였습니다.

## CI는 어떻게 동작하는가? (핵심 프로세스)

CI의 동작 방식은 다음과 같습니다.

1. **코드 커밋 및 푸시:** 개발자가 코드를 GitHub(중간 저장소)와 같은 중앙 저장소의 특정 브랜치에 커밋(Commit)하고 푸시(Push)합니다.
2. **자동 빌드 및 테스트:** 푸시가 감지되면 CI/CD 서버 (Jenkins, GitHub Actions 등)가 자동으로 빌드를 수행하고, 이어서 유닛 테스트와 통합 테스트를 실행합니다.
    - **빌드 단계에서 해결하는 문제:** 코드가 컴파일되지 않거나(컴파일 에러), 필요한 라이브러리가 누락되었거나, 여러 개발자의 코드가 병합될 때 발생한 충돌(Merge Conflict)로 인해 프로젝트가 완성된 형태로 만들어지지 않는 문제를 감지하고 알립니다.
    - **테스트 단계에서 해결하는 문제:** 빌드는 성공했지만, 코드의 기능이 예상대로 동작하지 않거나(로직 오류), 새로운 코드 변경으로 인해 기존 기능에 문제가 생기는(회귀 버그) 경우를 유닛 테스트 및 통합 테스트를 통해 찾아내고 알립니다.
3. **결과 알림:** 빌드 또는 테스트 결과가 통과하지 못할 경우, CI 시스템은 해당 개발자에게 Slack이나 이메일 등으로 실패를 즉시 알립니다.

## CI가 가져다주는 핵심 이점

CI의 도입은 개발 프로세스에 다음과 같은 주요 이점을 제공했습니다.

- **문제점 조기 발견:** 통합 과정의 오류를 초기에 발견하여 수정 비용과 시간을 절감했습니다.
- **개발 생산성 향상:** 자동화된 검증 과정을 통해 개발자들이 코드 작성에 더 집중할 수 있게 했고, 반복적인 오류 해결 작업 부담을 줄였습니다.
