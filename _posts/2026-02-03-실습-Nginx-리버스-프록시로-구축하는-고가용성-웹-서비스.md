안녕하세요! 오늘은 Docker 컨테이너와 Nginx를 활용해 리버스 프록시를 설정하고, 서버 장애 상황이나 설정 오류 시 시스템이 어떻게 반응하는지 다양한 시나리오로 테스트해 본 과정을 공유합니다.

## 1. 실습 개요 및 목표

- **목표:** Nginx의 리버스 프록시 기능을 활용해 2개의 웹 서버 컨테이너로 트래픽 분산하기
- **환경:** Docker, Nginx, Static HTML(web1, web2)

## 2. 시스템 구성

사용자가 브라우저를 통해 접속하면, Nginx 프록시 서버가 이 요청을 먼저 받습니다. Nginx는 설정된 규칙에 따라 뒤에 숨겨진 `web1`, `web2` 컨테이너 중 적절한 곳으로 트래픽을 전달합니다.

> **구조:** Client → Nginx Proxy (Port 80) → [ web1 (Port 80) / web2 (Port 80) ]
> 

---

## 3. 정상 동작 및 가중치(Weight) 테스트

먼저, 두 서버에 트래픽이 균등하게 가는지 확인한 후, 특정 서버에 더 많은 부하를 주는 **가중치 설정**을 진행했습니다.

### 설정 방법 (`nginx.conf`)



{% highlight bash %}
upstream backend_servers {
    server web1:80 weight=3;  # web1에 3번 보낼 때
    server web2:80 weight=1;  # web2에 1번 보냄
}

server {
    listen 80;
    location / {
        proxy_pass http://backend_servers;
    }
}
{% endhighlight %}
- **결과:** 실제로 요청을 보냈을 때 `web1`의 응답이 3번, `web2`의 응답이 1번 순차적으로 돌아오는 것을 확인했습니다. 서버 사양이 다른 경우 유용하게 쓰일 기능입니다.

## 4. [시나리오 A] 물리적 장애 발생 (Server Down)

가장 궁금했던 부분입니다. **"서버 하나가 죽으면 서비스는 중단될까?"**

### 실험 과정

1. `docker stop web1` 명령어로 서버 하나를 강제로 종료.
2. `curl`을 통해 지속적으로 요청 전송.

### 발견한 점과 고민

서버를 죽였음에도 불구하고 서비스는 끊기지 않고 `web2`의 내용을 가져왔습니다. 처음에는 **"Nginx가 타임아웃이 날 때까지 기다렸다가 넘겨주는구나"**라고 생각했습니다.

**하지만 공부해보니 다른 반전이 있었습니다!**

- 컨테이너가 **Stop** 된 상태라면 운영체제는 즉시 '연결 거부' 응답을 보냅니다.
- 사용자 입장에서 느껴진 미세한 지연은 타임아웃 때문이라기보다, Nginx가 `web1`의 에러를 인지하고 `web2`로 **재시도(Next Upstream)**를 하는 과정에서 발생한 시간이었던 것이죠.

## 5. [시나리오 B] 설정 오류 발생 (404 Not Found)

이번에는 `proxy_pass` 경로를 일부러 잘못 지정해 보았습니다.

- **현상:** `404 Not Found` 에러 발생.
- **분석:** 이 에러는 프록시 서버가 웹 서버(upstream)에 연결은 성공했지만, 요청한 경로에 파일이 없을 때 발생합니다. 즉, **Nginx와 웹 서버 간의 연결 자체는 물리적으로 문제가 없음**을 의미합니다.

## 6. 실습을 마치며: 운영 관점에서의 한 줄 평

이번 실습에서 사용한 핵심 명령어들의 의미를 다시 한번 정리해 봅니다.

| **명령어/옵션** | **기능** | **필요성** |
| --- | --- | --- |
| **`upstream`** | 부하 분산 대상 그룹 정의 | 여러 서버를 하나의 그룹으로 묶어 관리하기 위해 필수 |
| **`weight`** | 특정 서버 가중치 부여 | 서버 성능 차이가 있을 때 효율적인 자원 배분 가능 |
| **`proxy_pass`** | 요청 전달 | 클라이언트 요청을 실제 로직이 있는 서버로 넘겨주는 핵심 브릿지 |
