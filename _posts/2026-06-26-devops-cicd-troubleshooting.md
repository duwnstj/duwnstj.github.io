## 📌 TL;DR (3줄 요약)

* **Task:** Jekyll Chirpy 기반 정적 블로그의 GitHub Actions 자동 배포 파이프라인 에러 해결 및 네이버/구글 서치 콘솔 연동.
* **Action:** 도입하려던 Slack 웹훅 Step 제거로 CI/CD 정상화, 테마 버전에 따른 YAML 구조 변경 추적(`analytics: google:`), 로봇 캐싱 TTL을 고려한 파일 인증 적용.
* **Result:** CI/CD '안전망'을 통한 안정적인 무중단 배포 체계 구현 및 SEO 연동 완료.

---

## 1. Situation (상황)

블로그를 운영하기 위해 **GitHub Actions**를 도입했습니다. 
하지만 자동 배포(Deploy) 파이프라인이 중간 단계에서 계속 실패했고, 네이버 서치어드바이저와 구글 애널리틱스 연동 과정에서 사이트를 찾지 못하는 **404 에러 및 데이터 수신 실패 현상**이 발생했습니다.

---

## 2. Task (과제)

1. **CI/CD 파이프라인 정상화:** 끊어진 배포 파이프라인을 복구하여 코드 변경 사항이 프로덕션에 안전하게 도달할 수 있는 '안전망(Safety Net)' 구축.
2. **SEO 및 서드파티 연동 디버깅:** 외부 검색 로봇(Naver, Google)이 사이트 소유권을 정상적으로 인증하고 데이터를 수집할 수 있도록 설정 파일 수정.

---

## 3. Action (문제 해결 과정)

### A. CI/CD 파이프라인 복구: Slack 알림 스텝 제거

파이프라인에 Slack 알림 기능을 추가하려고 시도했으나, 에러 로그를 분석해보니 이 과정에서 문제가 발생했습니다.

* **문제 원인:** GitHub Repository의 `Secrets`에 Slack 웹훅 URL이 제대로 등록되지 않은 상태에서 해당 스텝이 실행되어 파이프라인 전체가 중단되었습니다.
* **해결 조치:** 현재 필수적이지 않은 Slack 알림 스텝을 과감히 제거하여 전체 CI/CD 파이프라인을 다시 정상화했습니다.

### B. 서드파티(SEO) 연동 트러블슈팅

#### ① 네이버 서치어드바이저의 404 캐싱 현상

네이버 소유권 확인용 HTML 파일을 배포한 후 터미널에서 `curl -I` 명령어로 확인했을 때는 `200 OK`가 떴지만, 네이버 로봇은 계속 `404 Not Found`를 띄웠습니다.

* **원인:** 네이버 검색 로봇 측의 **404 캐싱(TTL) 쿨타임** 때문이었습니다. 
* **해결:** 파일은 정상 배포되었음을 팩트로 확인했으므로, 인프라 측면의 캐싱 만료를 기다린 후 재요청하여 해결했습니다.

#### ② Jekyll Chirpy 테마의 Silent Failure (소리 없는 실패)

구글 애널리틱스(GA) 코드가 HTML에 삽입되지 않았고, 빌드 과정에서 아무런 에러 로그도 남지 않는 무시(Silent Failure) 현상이 발생했습니다.

* **원인:** 오픈소스 테마가 업데이트되면서 내부 스키마 구조가 변경된 것이 원인이었습니다.
* **해결:** 테마 문서를 추적하여 아래와 같이 계층 구조로 YAML 설정을 리팩토링했습니다.

```yaml
# 구버전 방식 (무시됨)
# google_analytics: "G-XXXXXXXXXX"

# 신버전 방식 (정상 작동)
analytics:
  google:
    id: "G-XXXXXXXXXX"
```

---

## 4. Result (결과)

* **안전망 확보:** 마크다운 오타나 문법 실수를 내더라도 GitHub Actions의 **Build 단계에서 즉시 차단**됩니다. 잘못된 코드가 유저에게 노출되는 것을 차단하는 무중단 배포 체계를 완성했습니다.

---

## 💡 회고 및 Lessons Learned (배운 점)

이번 트러블슈팅을 통해 단순한 블로그 구축을 넘어, 인프라 엔지니어링 관점의 중요한 두 가지 레슨을 얻었습니다.

1. **CI/CD 파이프라인은 귀찮은 작업이 아니라 '안전망'이다**
   로컬에서 매번 테스트하고 푸시하면 당장은 편할 수 있습니다. 하지만 사람이 하는 일이기에 마크다운 문법이나 설정 파일에 실수를 할 수밖에 없습니다. CI/CD를 도입함으로써, 코드에 문제가 생기면 GitHub Actions가 빌드를 멈추고 **잘못된 결과물이 라이브 서버에 배포되는 것을 원천 차단**하는 강력한 보안망을 경험했습니다.

2. **Silent Failure(조용한 에러) 디버깅의 기본은 '원형 팩트 체크'**
   테마 설정 오류처럼 에러 로그조차 남지 않는 상황에서는 당황하기 쉽습니다. 하지만 터미널에서 `curl` 명령어로 실제 서버에 렌더링된 HTML 소스코드를 먼저 뜯어봄으로써, '스크립트 자체가 주입되지 않았다'는 명확한 팩트를 확인할 수 있었습니다. **추측하기 전에 로그와 원본 데이터를 먼저 확인하는 습관**의 중요성을 깨달았습니다.
