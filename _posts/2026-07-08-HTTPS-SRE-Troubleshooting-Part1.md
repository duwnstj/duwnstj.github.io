# 🚀 AWS SRE 대서사시 1/4: Terraform 파괴의 나비효과

> 이 글은 인프라 구축부터 CI/CD 배포까지 이어지는 SRE 트러블슈팅 대서사시의 첫 번째 파트입니다.
> - **[1/4] Terraform 파괴의 나비효과: VPC 철거 딜레이와 소크라테스 디버깅** (현재 글)
> - [2/4] DNS 권한 위임과 ACM 전파 지연 트러블슈팅
> - [3/4] Terraform State 기억상실증과 Import 복구기
> - [4/4] 무중단 배포의 덫: GitHub Actions와 CodeDeploy 캐시 트러블슈팅

## ⏱️ 1. 10초 요약
- **문제**: `terraform destroy` 실행 중 6분 넘게 멈춰있는 것을 참지 못하고 `Ctrl+C`로 강제 종료함. 이후 재실행 시 `DependencyViolation: Network vpc-... has some mapped public address(es)` 에러가 발생하며 VPC와 IGW 철거에 실패함.
- **해결**: AI 튜터와의 대화를 통해 강제 종료가 상태 불일치를 일으켰음을 깨닫고, AWS 콘솔에서 수동으로 VPC 삭제를 시도하며 백그라운드 찌꺼기(네트워크 인터페이스)가 소멸되는 것을 확인하여 철거를 완료함.
- **핵심 깨달음**: 클라우드 인프라(특히 DB, ALB)는 삭제 명령이 떨어져도 안전한 연결 종료(Draining)와 백업 때문에 수 분의 물리적 시간이 소요된다. **절대 중간에 강제 종료해서는 안 된다.**

## 🏗️ 2. 아키텍처 진화
- 이전까지는 리소스를 생성(`apply`)하는 것에만 집중했다면, 이번 트러블슈팅을 통해 **안전하게 리소스를 파괴(`destroy`)하는 SRE 관점의 라이프사이클**을 이해하게 됨.
- Terraform의 `tfstate` 상태 관리와 실제 AWS 물리 리소스 간의 "상태 불일치(Drift)"가 어떤 장애를 유발하는지 아키텍처 레벨에서 경험함.

## 🔍 3. Deep Dive: 왜 대문을 못 부수는가?
`terraform destroy`를 실행하면 테라폼은 의존성 역순으로 리소스를 삭제한다. EC2/RDS ➔ 서브넷 ➔ IGW ➔ VPC 순서다.
하지만 도중에 `Ctrl+C`를 누르면, 백그라운드에서는 아직 리소스가 지워지는 중인데 테라폼 기사는 퇴근해버린다. 다시 `destroy`를 누르면 테라폼은 족보가 꼬인 상태에서 다짜고짜 IGW(대문)부터 부수려고 시도하게 되고, AWS API는 **"VPC 안에 아직 Public IP를 쥐고 있는 녀석(의존성)이 남아있다!"** 며 `DependencyViolation` 에러를 뱉어내고 철벽을 쳐버린다.

## 🗣️ 4. 소크라테스 디버깅 일지 (AI와의 치열한 티키타카 및 사고 과정)

**[사건의 발단: 기다림의 미학을 참지 못한 자]**
- **나**: "아니, `terraform destroy` 쳤는데 6분 10초 동안 멈춰 있길래 `Ctrl+C` 해서 나왔거든? 그래서 그런 건가? 또 VPC 에러야. 이거 그냥 '승인' 치트키 써서 강제로 지우게 할 순 없어?"
- **AI 튜터**: "클라우드 세계에선 '내가 만들었으니 묻지 말고 지워줘!'라는 치트키는 없습니다. 건물주(AWS)는 건물 안에 의자 하나라도 남아있으면 절대 대문을 못 부수게 합니다. 직접 콘솔 VPC 탭에 가서 삭제를 눌러보며 숨어있는 찌꺼기의 정체를 파악해 보세요!"

**[Aha-Moment: 아! 이게 이런 구조였구나!]**
- **나**: "어? 저 VPC를 수동으로 삭제 버튼 누르자마자 destroy가 끝났는데?"
- **나의 깨달음 (Aha-Moment)**: "아! 테라폼이 멈춰있던 게 아니라, **무거운 리소스(RDS 등)가 안전하게 삭제되느라 물리적인 시간(Draining)이 필요했던 거구나.** 내가 콘솔에서 삭제를 시도하는 그 찰나의 순간에 백그라운드에서 늦게 지워지던 네트워크 선이 마침내 다 뽑힌 거였어! 테라폼은 그냥 무작정 기다리고 있었던 건데 내가 성질 급하게 포크레인 기사를 쫓아냈던 거네!"

## 💡 5. 파인만 비유 부록
- **포크레인과 대문**: 우리가 식당(VPC)을 철거하려고 포크레인(Terraform)을 불렀다. 기사가 "1. 식당 안의 집기 빼기 ➔ 2. 전화선 뽑기 ➔ 3. 대문 부수기" 순서로 작업 중인데, 사장님(나)이 6분 만에 답답하다고 "기사 당장 퇴근해!"(`Ctrl+C`) 해버림. 
- **꼬여버린 족보**: 다시 기사를 부르니, 기사는 족보가 꼬여서 다짜고짜 대문부터 부수려 함. 하지만 건물주(AWS)가 "야! 안에 아직 직통 전화선 안 뽑았어!" 라며 철거를 막아선 상황.

## ⚖️ 6. Trade-off (의사결정)
- **강제 종료(Ctrl+C) vs 기다림(Wait)**
  - **강제 종료**: 콘솔이 멈춘 것 같아 답답함을 즉시 해소할 수는 있지만, Terraform 상태 파일(`tfstate`)과 클라우드의 실제 물리 상태 간의 심각한 불일치(Drift)를 유발하여 디버깅 비용이 배로 듦.
  - **기다림 (선택)**: 인프라 파괴는 복구 불가능한 작업이므로 안전한 종료 절차(스냅샷, 커넥션 드레이닝 등)를 위해 10분 이상의 인내심이 필수적이다. SRE는 기다림을 감수해야 연재를 이어나갈 수 있다.

## ⭐ 7. STAR-F 면접 방어 Q&A

**Q. 클라우드 인프라 철거 시 Terraform이 멈춘 것처럼 보일 때 어떻게 대처하시겠습니까?**
- **Situation**: 인프라 비용 절감을 위해 `terraform destroy`를 실행했으나 5분 이상 프롬프트가 응답하지 않는 상황이 발생했습니다.
- **Task**: 안전하게 클라우드 리소스를 파괴하고 Terraform 상태의 정합성을 유지해야 했습니다.
- **Action**: 처음에는 멈춘 줄 알고 프로세스를 강제 종료(Ctrl+C)하여 상태 불일치 에러(`DependencyViolation`)를 겪었습니다. 이후 이 현상이 에러가 아니라, 로드밸런서나 RDS 같은 리소스가 연결을 정리(Draining)하거나 스냅샷을 생성하는 데 필요한 '물리적 소요 시간'임을 깨달았습니다. 에러 해결을 위해 AWS 콘솔에서 잔존 의존성을 확인하며 백그라운드 작업이 끝날 때까지 기다린 후, 다시 명령을 실행하여 꼬인 상태를 복구했습니다.
- **Result**: 이를 통해 클라우드 리소스의 라이프사이클과 의존성 관계를 깊이 이해하게 되었으며, 이후로는 인프라 작업 시 강제 종료를 피하고 CloudWatch나 콘솔을 통해 백그라운드 진행 상태를 모니터링하는 SRE 습관을 갖게 되었습니다.
- **FinOps/SRE Point**: 리소스가 완전히 삭제되지 않고 찌꺼기(EIP, NAT, 스냅샷 등)가 남을 경우 발생하는 **'좀비 과금'**을 방지하기 위해, 인프라의 완전한 소멸(Zero State)을 확인하는 것이 FinOps의 핵심입니다.
