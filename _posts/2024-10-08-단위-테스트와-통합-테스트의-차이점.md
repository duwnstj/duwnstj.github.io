- 단위 테스트와 통합 테스트는 소프트웨어 테스트에서 중요한 두가지 종류의 테스트입니다. 두 테스트는 테스트의 범위와 목적이 다르며, 각각의 장단점이 있습니다.

### 1. 단위 테스트(Unit Test)

- **정의** : 단위 테스트는 애플리케이션의 작은 단위 , 즉 메서드나 함수와 같은 개별적인 모듈을 테스트하는 방법입니다. 외부 의존성을 Mocking 하거나 Stubbing하여 개별 모듈의 동작을 검증합니다. 

- **주요 목적** : 코드의 각 단위가 올바르게 동작하는지 확인하는 것

- **장점** : 

- 빠른 테스트 실행 속도 : 외부 의존성이 없거나 Mocking으로 대체되기 때문에 테스트 실행이 매우 빠릅니다.
- 버그 발견이 용이 : 작은 단위로 테스트하므로 버그가 발생한 부분을 정확하게 추적하기 쉽습니다.
- 코드 품질 향상 : 세부적인 동작을 검증하므로 코드 품질을 개선하고 유지보수성을 높입니다.

- **단점** : 
- 제한된 범위 : 개별 단위의 동작만 검증하므로 시스템 전체의 동작을 보장하지는 못합니다.
- Mock 객체의 과도한 사용: 많은 외부 의존성을 Mock으로 처리하다 보면 실제 환경과는 다른 결과가 나타날 수 있습니다.

### 2. 통합 테스트(Integration Test)
- **정의** : 통합 테스트는 여러 모듈이 상호 작용하는 방식이나 외부 시스템과의 통합이 올바르게 이루어지는지 검증하는 테스트입니다. 데이터베이스, 네트워크, 파일 시스템 등 실제 의존성들을 포함해 전체 시스템의 동작을 확인합니다.

- **주요 목적** : 여러 모듈 간의 통합과 상호작용이 예상대로 동작하는지 확인하는것

- **장점** : 
- 실제 환경에 가까운 테스트 : 실제 의존성을 사용하여 더 현실적인 시나리오를 테스트하므로 실제 환경에서 발견할 수 있는 문제를 발견할 수 있습니다. 
- 전체적인 안정성 검증 : 시스템 간 상호작용이 예상대로 작동하는지 확인할 수 있어,전체 시스템의 안정성을 보장할 수 있습니다. 

- **단점** : 

- 느린 실행 속도 : 외부 시스템과의 통합을 포함하므로 단위 테스트보다 훨씬 느립니다.
- 버그 발견이 어려움 : 버그가 발생했을 때 여러 모듈이 연관되어 있기 때문에 정확한 문제의 원인을 찾는데 시간이 오래 걸릴 수 있습니다.

## 꼬리 질문 예상 

### 1. 단위 테스트와 통합 테스트는 언제 각각 사용해야하나요?

- **단위 테스트**는 빠른 피드백이 필요한 경우, 그리고 코드의 세부적인 로직을 검증할 때 사용합니다. 개발자가 코드를 자주 변경할 때마다 단위 테스트를 통해 변경 사항이 기존 기능에 영향을 미치지 않는지 확인 할 수 있습니다.
- **통합 테스트**는 시스템 전체의 동작을 검증하거나 외부 시스템과의 통합을 확인하고자 할 때 사용합니다. 일반적으로 배포 전에 통합 테스트를 통해 시스템이 예상대로 동작하는지 검증합니다. 

### 2. 단위 테스트에서 Mock을 사용하는 이유는 무엇인가요?
- 단위 테스트의 목적은 코드의 작은 단위를 독립적으로 검증하는 것이기 때문에 외부 시스템이나 의존성으로부터 테스트 환경을 격리하는 것이 중요합니다. Mock 객체를 사용하여 이러한 외부 의존성을 대체하면, 실제 환경과는 분리된 상태에서 코드의 동작만을 검증할 수 있습니다.

### 3. 단위 테스트와 통합 테스트 중 어느 것이 더 중요한가요?
- **둘다 중요합니다.** 단위 테스트는 빠르게 피드백을 제공하고 코드의 품질을 개선하는데 필수적이며 통합테스트는 전체 시스템이 문제 없이 동작하는지를 확인 할 수 있기 때문에 시스템의 안정성을 보장합니다. 
- 두가지 테스트를 함께 사용하여 다양한 시나리오에서 문제를 사전에 발견하고 품질을 높일 수 있습니다.

### 4. 통합 테스트에서 가장 흔히 발생하는 문제는 무엇인가요?
- 통합테스트에서 가장 흔한 문제는 **환경 의존성**입니다. 테스트 환경과 실제 운영환경이 다를 경우 , 테스트는 성공했지만 실제 운영에서 실패하는 경우가 발생할 수 있습니다. 이를 방지하기 위해 가능한 한 운영환경과 유사한 환경에서 통합 테스트를 실행하는 것이 중요합니다.
