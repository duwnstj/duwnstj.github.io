스프링 부트로 API를 개발하면서 가졌던 근본적인 의문들이 있었습니다. "왜 내장 서버가 있는데 아파치를 따로 둘까?", "트래픽은 어떻게 분산될까?" 이 고민들을 하나씩 풀어가며 정리한 인프라의 핵심 구조입니다.

### 1. 프록시(Proxy)가 필요한 이유

단순히 서버가 한 대라면 직접 연결하면 그만이지만, 서비스 규모가 커지면 관리가 불가능해집니다. 이때 **중계기이자 다리 역할**을 하는 프록시 서버가 필수적입니다.

- **고민의 지점:** 직접 연결하지 않고 프록시를 거치는 이유는 무엇일까?
- **정리:** 클라이언트 요청이 다이렉트로 가는 것이 아니라 프록시를 거치며 분산되기 때문입니다.
    - **포워드 프록시:** 인터넷 앞단에서 요청을 전달하며, 서버가 사용자의 정보를 알 수 없게 보호합니다.
    - **리버스 프록시:** 인터넷 뒷단에서 서버로 요청을 전달하며, 사용자가 서버의 정보를 알 수 없게 감춥니다.

### 2. 라운드 로빈(Round Robin)의 한계와 보완

트래픽을 분산할 때 가장 기본이 되는 알고리즘은 **라운드 로빈**입니다. 요청을 순차적이고 균등하게 나눠주는 방식이죠. 하지만 여기서 실무적인 한계가 발생합니다.

- **문제 상황:** 만약 서버 A는 노후되었고 서버 B는 신설 서버라면?
- **결과:** 똑같이 100개의 요청을 보냈을 때, 저성능 서버 A는 부하를 견디지 못하고 응답이 느려지다 결국 죽게 됩니다.
- **대안:** 그래서 **HAProxy** 같은 설정에서는 **'서버 가중치'**를 두어 트래픽 분산을 조절합니다. 라운드 로빈을 기본으로 하되, 다른 알고리즘을 추가해 보완하는 것이 핵심입니다.

### 3. AWS 로드 밸런싱과 관리의 진화

과거에는 HAProxy의 설정 파일(`config`)을 직접 수정해야 했지만, 이제는 **AWS 로드 밸런서**를 통해 이 과정이 API 호출 방식으로 추상화되었습니다.

- **인프라의 변화:** 파일 수정 방식에서 API 기반으로 진화하며 **쿠버네티스** 같은 도구로 통합 모니터링과 제어가 가능해졌습니다.
- **오토 밸런싱:** 헬스 체크를 통해 서버를 중앙 관리하고, 트래픽에 따라 서버를 자동으로 늘리거나 줄입니다.
- **주의점:** 무분별한 서버 증설 설정은 비용 폭탄으로 이어질 수 있으므로 운영 관점에서의 주의가 필요합니다.

### 4. 아파치와 톰캣: API 호출의 연결 고리

"내장 톰캣이 있는데 아파치가 왜 또 필요할까?"라는 질문의 답은 역할 분담에 있었습니다. 아파치는 단순한 문지기를 넘어, 톰캣이 일을 할 수 있도록 중요한 정보들을 실어 나릅니다.

- **아파치(Apache):** 외부 요청을 받아 API 서버(톰캣)로 연결해주는 **리버스 프록시**입니다.
- **톰캣(Tomcat):** 아파치가 넘겨준 요청을 받아 실제 비즈니스 로직을 처리하고 **JSON 데이터**를 응답합니다.

### **💡 아파치가 톰캣에게 넘겨주는 핵심 정보들**

아파치는 리버스 프록시 역할을 수행하며 다음과 같은 데이터들을 톰캣에 전달합니다. 이 정보들이 있어야 비로소 온전한 API 호출이 가능해집니다.

1. **사용자 실제 IP (`X-Forwarded-For`):** 프록시를 거치면 서버는 프록시의 IP만 보게 됩니다. 아파치는 헤더에 사용자의 진짜 IP를 담아 전달하여 서버가 접속자를 식별하게 돕습니다.
2. **HTTP 헤더와 인증 정보:** 클라이언트가 보낸 쿠키, 인증 토큰(JWT), 컨텐츠 타입 등을 그대로 밀어 넣어줍니다.
3. **프로토콜 정보 (`X-Forwarded-Proto`):** 사용자가 HTTPS로 안전하게 들어왔는지 정보를 넘겨주어 보안 처리를 가능하게 합니다.
4. **요청 본문(Body):** 사용자가 보낸 실제 JSON 데이터 등을 통째로 전달합니다.

**결국 아파치가 요청 트래픽과 상세 정보들을 해당 API로 정확히 프록시해주기 때문에, 우리가 개발한 API 호출이 비로소 가능해지는 것입니다.**
