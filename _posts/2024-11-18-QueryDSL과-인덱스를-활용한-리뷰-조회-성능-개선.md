## 개요 
리뷰 조회 로직에서 발생한 성능 문제를 해결하기위해 데이터베이스 인덱스를 설계하고 테스트를 해보았습니다.

## 1. 성능 테스트 환경

### 테스트 데이터 준비
- **총 데이터 개수** : 100,000개
- **데이터 분배**: 
`bookId`와 `userId`를 각각 1~6으로 균등하게 분배
`reviewStar`는 1~5점으로 설정했으며 중복 데이터가 포함되어 있습니다.

### 테스트 시나리오
- **도구** : JMeter
- **조건** : 
    - 100명의 사용자가 5초마다 `reviewStar` 조건으로 리뷰를 조회합니다.
    - 데이터베이스와 API의 응답 속도 및 처리량을 측정하였습니다.
    
## 기존 Review 코드

```java

@Entity
@Getter
@NoArgsConstructor
@Table(name = "review")
public class Review extends Timestamped {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private Integer reviewStar;
    @Column(nullable = false)
    private String reviewTitle;
    @Column(nullable = false)
    private String reviewDescription;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "book_id")
    private Book book;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User user;

    public Review(Integer reviewStar, String reviewTitle, String reviewDescription, Book book, User user) {
        this.reviewStar = reviewStar;
        this.reviewTitle = reviewTitle;
        this.reviewDescription = reviewDescription;
        this.book = book;
        this.user = user;
    }

    public void update(Integer reviewStar, String reviewTitle, String reviewDescription) {
        this.reviewStar = reviewStar;
        this.reviewTitle = reviewTitle;
        this.reviewDescription = reviewDescription;
    }
}
```text
## 2. 성능 개선 목표 
기본적으로 제공되는 PK 인덱스(`userId`,`bookId`)만으로는 특정 별점(`reviewStar`)조건을 조회할 때 성능이 저하되었습니다.

### 문제점
- 별점(`reviewStar`) 조건으로 데이터를 조회할 때 **Full Table Scan**이 발생
- 이는 대량 데이터에서 성능 저하로 이어질 가능성이 높습니다.

### 개선 목표
- `reviewStar`와 `bookId`를 활용해 효율적인 쿼리를 설계합니다.
- 인덱스를 추가하여 응답 시간 단축과 처리량(TPS) 증가를 목표로 합니다.

## 3. 인덱스 설계 및 적용
### 기존 상태 : 기본 PK 인덱스
- `userId`와 `bookId`는 Primary Key로 자동 인덱스가 설정되어 있었습니다.
- 하지만 자주 사용되는 조회 조건인 `reviewStar`에는 인덱스가 적용되어있지 않았습니다

### 개선 방안 : 단일 및 복합 인덱스 추가
1. 단일 인덱스
    - `reviewStar`에 단일 인덱스를 추가하여 특정 별점 조건의 조회 성능을 개선합니다.
2. 복합 인덱스
    - `bookId,reviewStar` 복합 인덱스를 추가하여 `bookId`로 데이터를 좁힌 후 `reviewStar`로 추가 필터링을 수행합니다.

### 인덱스 적용 후
```java
@Entity
@Getter
@NoArgsConstructor
@Table(
        //별점에 단일 인덱스 적용,bookid와 별점 복한 인덱스 적용
        name = "review",
        indexes = {
                @Index(name = "idx_review_star", columnList = "reviewStar"),
                @Index(name = "idx_book_review_star", columnList = "book_id,reviewStar")
        }
)
public class Review extends Timestamped {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private Integer reviewStar;
    @Column(nullable = false)
    private String reviewTitle;
    @Column(nullable = false)
    private String reviewDescription;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "book_id")
    private Book book;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User user;

    public Review(Integer reviewStar, String reviewTitle, String reviewDescription, Book book, User user) {
        this.reviewStar = reviewStar;
        this.reviewTitle = reviewTitle;
        this.reviewDescription = reviewDescription;
        this.book = book;
        this.user = user;
    }

    public void update(Integer reviewStar, String reviewTitle, String reviewDescription) {
        this.reviewStar = reviewStar;
        this.reviewTitle = reviewTitle;
        this.reviewDescription = reviewDescription;
    }
}
```text

## 5. 인덱스가 필요한 이유

### Full Table Scan 방지
- 인덱스를 추가하지 않으면 별점 조건으로 데이터를 조회할 때 **모든 데이터를 스캔**해야합니다.
- 이는 대량 데이터 환경에서 **성능 병목 현상(성능 제한)**을 유발할 가능성이 높습니다.

### 자주 사용되는 조건
- `reviewStar`는 자주 사용되는 필터 조건으로 단일 인덱스만으로도 쿼리 속도를 향상시킬 수 있습니다.

### 데이터 분포의 중요성
- 인덱스의 효율은 데이터의 분포에 따라 달라집니다.
- 테스트 데이터는 분포가 고르게 설정되어 최적화 효과가 제한적일 수 있습니다.
- 실제 서비스 환경에서는 데이터 분포가 편향될 가능성이 높아, 더 큰 성능 개선 효과를 기대할 수 있습니다.

## 6. ExPLAIN을 활용한 쿼리 최적화

### 실행 계획 확인
MySQL의 `EXPLAIN` 명령어를 사용하여 쿼리가 인덱스를 제대로 활용하는지 확인하였습니다.

```sql
EXPLAIN SELECT * 
FROM review
WHERE book_id = 1 AND review_star = 4;

```text
