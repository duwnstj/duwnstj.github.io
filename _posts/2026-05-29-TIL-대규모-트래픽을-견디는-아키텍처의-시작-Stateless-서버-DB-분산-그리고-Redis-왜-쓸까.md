## 🚀 시작하며

프로젝트를 혼자 개발하거나 로컬 환경에서 테스트할 때는 보통 `서버 1대 + DB 1대` 구조로 시작합니다. 이때는 시스템이 아주 단순하고 잘 작동하죠. 하지만 사용자가 수만 명, 수십만 명으로 늘어나는 **대규모 운영 환경**이 되면 이 단순한 구조는 여지없이 무너집니다.

오늘 공부하며 "왜 대형 서비스들이 서버를 스테이트리스(Stateless)로 만들고, DB를 여러 대로 쪼개고, 굳이 Redis를 중간에 끼워 넣는지" 그 유기적인 흐름과 원리를 깨닫게 되어 이를 정리해 봅니다.

---

## 1. 서버를 가볍게: 스테이트리스(Stateless)와 수평 확장(Scale-Out)

대규모 트래픽이 몰리면 서버 1대로는 감당할 수 없어 똑같은 서버를 여러 대로 늘리는 **수평 확장(Scale-Out)**을 해야 합니다. 이때 핵심이 되는 개념이 바로 **Stateless(상태 없음)**입니다.

- **Stateful (상태 있음):** 서버 메모리에 사용자의 로그인 세션 정보를 저장합니다. 이 경우, 사용자의 다음 요청이 다른 서버로 가면 로그인 정보가 없어 로그아웃되는 대참사가 일어납니다. 서버를 마음대로 늘리기 어렵습니다.
- **Stateless (상태 없음):** 서버 자체 메모리에는 사용자의 상태를 절대로 저장하지 않습니다. 대신 클라이언트가 '인증 증명서(Token/JWT)'를 매 요청마다 들고 다니게 하거나, 외부 공용 저장소를 바라보게 합니다.

> **💡 핵심 깨달음:** 서버를 Stateless하게 만들면 어떤 서버가 요청을 받아도 똑같이 처리할 수 있으므로, 트래픽에 따라 서버를 10대든 100대든 아무 걱정 없이 복사하듯이 늘릴 수(Scale-Out) 있습니다.
> 

---

## 2. DB도 팀플레이가 필요하다: 읽기/쓰기 분리(Replication)

서버를 Stateless로 만들어서 여러 대로 늘렸더니, 이제는 모든 서버가 단 하나의 DB로 몰려가서 문을 두드리기 시작합니다. 결국 **DB 병목 현상(Bottleneck)**이 발생하죠. 이 문제를 해결하기 위해 DB도 역할을 나누어 클러스터(분산 구조)로 관리해야 합니다.

현업 대규모 아키텍처에서는 DB를 **Source(Master)와 Replica(Slave)** 구조로 분류하여 역할을 맡깁니다.

| DB 유형 | 주 역할 | 허용 쿼리 | 특징 |
| --- | --- | --- | --- |
| **Source DB** | 데이터 원본 관리 | 쓰기/수정/삭제 (`Insert, Update, Delete`) | 오직 1대만 존재하며, 데이터의 정합성을 책임짐 |
| **Replica DB** | 데이터 복제본 | 오직 조회 (`Select`) | 여러 대로 늘릴 수 있으며, 대부분의 트래픽인 '조회'를 분산 처리함 |
- **Application 레벨에서의 제어 (Spring Boot 예시):**
Spring Boot에서는 `@Transactional(readOnly = true)` 옵션을 주면, 프레임워크가 이를 인지하고 알아서 한가한 읽기 전용(Replica) DB로 쿼리를 라우팅해 줍니다.

---

## 3. 그런데 발생한 기괴한 버그: 복제 지연(Replication Lag)

DB를 분산해 두었을 때 트래픽이 극도로 몰리면 아주 신기한 장애가 발생할 수 있습니다.

> *"방금 로그인을 성공했는데, 마이페이지로 이동하자마자 로그아웃이 돼요. 그런데 새로고침을 몇 번 누르니까 다시 로그인 상태로 나와요!"*
> 

이 현상의 범인은 바로 **복제 지연(Replication Lag)**입니다.
사용자가 로그인을 하면 그 정보는 **Source DB**에 저장됩니다. 그리고 즉시 마이페이지를 조회하면 Spring Boot는 조회 요청이므로 **Replica DB**를 바라보게 하죠.

하지만 Source DB의 데이터가 Replica DB로 동기화되는 데 아주 미세한 시간(예: 0.5초)이 걸린다면? 동기화가 되기 전 찰나의 순간에 Replica DB를 조회한 사용자는 로그인 정보가 없다고 판단되어 로그아웃 처리가 되는 것입니다.

---

## 4. 구원투수 등판: 이래서 Redis를 쓰는구나!

이 '복제 지연' 문제를 원천 차단하고, DB의 부담을 획기적으로 줄이기 위해 도입하는 것이 바로 **Redis(In-Memory Cache)**입니다.

- **세션 데이터의 격리:** 로그인 상태나 세션처럼 실시간성이 극도로 중요하고 자주 바뀌는 데이터는 디스크 기반의 RDB가 아니라, **빛의 속도로 작동하는 메모리 기반 저장소인 Redis에 저장**합니다.
- **결과:** 10대의 서버가 동시에 Redis를 바라보아도 복제 지연 없이 언제나 동일하고 최신화된 로그인 상태를 보장받을 수 있습니다. DB는 더 이상 실시간 세션 관리에 자원을 낭비하지 않고, 진짜 중요한 결제나 회원가입 같은 핵심 비즈니스 데이터 처리에만 집중할 수 있게 됩니다.

## 🎯 마치며: 원리를 아는 엔지니어의 시각

단순히 "남들이 쓰니까", "유행하니까" Redis를 붙이고 DB 리플리케이션을 하는 게 아니었습니다.

1. 대규모 트래픽을 받아내기 위해 서버를 **Stateless**하게 늘렸고,
2. 그로 인해 대두된 DB 병목을 해결하고자 **DB 역할을 읽기/쓰기로 분리**했으며,
3. 분산 구조에서 필연적으로 발생하는 복제 지연과 속도 한계를 극복하기 위해 **Redis라는 핵심 인프라를 도입**하게 되는 일련의 유기적인 인프라 흐름을 이해하게 되었습니다.

앞으로 토이 프로젝트를 하더라도 "이 구조가 대규모 환경으로 가면 어떤 병목이 생길까?"를 항상 고민하는 엔지니어가 되어야겠습니다.
