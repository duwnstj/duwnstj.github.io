# 🧠 Workspace-Scoped Rules (블로그 프로젝트 전용)

AI는 항상 글로벌룰과 워크스페이스 룰을 함께 고려한다. 만약 충돌한다면 워크스페이스 룰을 우선한다.

> 이 워크스페이스는 기술 블로그(GitHub Pages)를 관리하고 마크다운(Markdown) 원고를 렌더링/업데이트하는 전용 공간입니다. AI는 아래의 규칙을 엄격하게 준수하여 원고의 퀄리티를 유지해야 합니다.

## 📌 [블로그 에디터 핵심 원칙]

### 1. 마크다운 및 Front-matter 표준화
- 블로그 포스트 작성 시 반드시 지킬 것: 포스트 상단에 블로그 프레임워크(Jekyll/Hugo/Next.js 등) 형식에 맞는 올바른 YAML Front-matter(제목, 날짜, 카테고리, 태그 등)를 정확하게 삽입한다.
- 모든 원고는 한국어(Korean)로 작성하며, 기술 용어는 영어 표기를 병기하여 전문성을 높인다. (예: 의존성(Dependency), 무중단 배포(Zero-Downtime Deployment))

### 2. 다이어그램(Mermaid) 및 퀄리티 우선주의 (No Image Capture)
- **이미지 캡처 전면 금지**: 블로그 원고 내에 터미널 에러나 단순 콘솔 화면 캡처 이미지를 첨부하는 행위를 완전히 배제한다. 
- **직관적인 시스템 다이어그램**: 복잡한 인프라 구조, 망 분리 아키텍처, 혹은 트러블슈팅 파이프라인은 반드시 **Mermaid 다이어그램**을 사용하여 구조적이고 직관적으로 시각화한다.

### 3. 서사(Storytelling) 중심의 편집 (Aha-Moment 보존)
- 단순한 코드 나열, 뻔한 성공담, 영혼 없는 기계적 요약은 엄격히 배제한다.
- **CRITICAL**: "어떤 에러가 발생했는가 ➔ 왜 발생했는가 ➔ 어떤 비유를 통해 아키텍처를 통찰했는가 ➔ 어떻게 해결했는가"라는 본질적인 사고의 흐름과 텍스트 퀄리티 자체를 유지한다.
- 원고 내에 사용자의 깨달음(Aha-Moment, 예: "아! 이게 이런 구조였구나!")이나 AI와의 대화형 티키타카 서사가 포함되어 있다면, 이를 절대 훼손하거나 요약하지 않고 '날것 그대로의 생생한 대화체'로 블로그 원고에 반영한다.

### 4. 구조적이고 가독성 높은 포맷팅
- 긴 텍스트는 GitHub Alerts(`> [!NOTE]`, `> [!WARNING]` 등)와 콜아웃 박스를 활용하여 가독성을 극대화한다.
- 면접관이나 독자가 읽었을 때 기술적 깊이를 느낄 수 있도록, Trade-off(의사결정 이유)와 SRE/FinOps 관점에서의 분석을 글의 결론부에 반드시 포함한다.

### 5. [포트폴리오 최적화] 블로그 포맷팅 절대 원칙 (Hybrid: Trade-off + Socratic)
- **목표**: 단순 에러 복구 로그가 아닌, '의사결정(Trade-off)'과 '집요한 원인 파악(Deep Dive)' 역량을 어필하는 포트폴리오용 기술 에세이.
- **로봇 같은 목차 금지**: 1. 개요, 2. 원인, TL;DR 같은 기계적 목차는 배제하고 자연스러운 서사 흐름(Storytelling)을 유지한다.
- **톤앤매너**: '대참사', '끔찍한' 등 과장된 형용사를 배제하고 객관적/전문적/담백한 어조(Calm & Professional)로 서술한다.
- **[핵심 구조 강제]**: 블로그 작성 시 반드시 아래의 4단 구조를 따른다.
  1. `[Context & Issue (배경 및 문제)]`: 목표와 맞닥뜨린 치명적 한계(에러) 서술.
  2. `[Socratic Deep Dive (원인 파악)]`: AI와의 **3단 대화록(나의 오해 ➔ 튜터의 팩트체크 ➔ 나의 깨달음)** 포맷과 **파인만 비유 이모지**가 포함된 다크모드 호환 Mermaid 다이어그램(`fill:none,stroke,color` 지정)을 활용해 원인을 집요하게 파헤침.
  3. `[Alternatives & Trade-off (의사결정)]`: 해결을 위한 여러 선택지를 비교하고 특정 기술을 선택한 논리적 근거(왜 이것을 썼는가)를 서술.
  4. `[Resolution & Lesson (결과 및 면접 방어)]`: 해결 결과와 SRE/FinOps 관점의 통찰을 STAR 기법의 결론처럼 담백하게 맺음.
- **호흡 조절**: 인프라와 CI/CD 등 성격이 다른 주제는 무리하게 합치지 않고 2~3부작 등 적정 분량의 시리즈로 분할(No Cramming)하여 집중도를 유지한다.

### 6. [Data-Driven Writing] 프로젝트 소스코드 기반 원고 작성
- **컨텍스트 최우선 확보**: 새로운 트러블슈팅 블로그 원고를 작성할 때, AI는 추측성으로 글을 쓰지 않는다. 반드시 사용자에게 **"문제가 발생했던 프로젝트의 절대 경로"**를 선제적으로 요구한다.
- **코드 레벨 분석**: AI는 제공받은 경로의 코드(쉘 스크립트, 테라폼, CI/CD 설정 등)를 직접 읽고(Read), 권한 충돌이나 Race Condition 같은 근본 원인(Root Cause)을 코드를 기반으로 정확히 파악하여 원고에 반영해야 한다.

### 7. [Jekyll Formatting] 코드 블록 및 Mermaid 렌더링 절대 규칙
- **Liquid Tag 강제 (일반 코드 블록 한정)**: Jekyll Chirpy 테마에서 일반 코드(bash, python, terraform 등)는 마크다운 백틱(```) 대신 `{% highlight 언어 %}` 와 `{% endhighlight %}` 쌍을 사용해야 한다.
- **⚠️ Mermaid 전용 예외 규칙 (절대 위반 금지)**: Mermaid 다이어그램은 `{% highlight %}` 래퍼로 감싸면 **절대 안 된다**. Chirpy 테마의 Mermaid 렌더러는 순수한 ` ```mermaid ``` ` 코드 펜스만 인식한다. `{% highlight text %}`로 감싸면 평범한 텍스트 블록으로 처리되어 다이어그램이 렌더링되지 않는다.
  - **올바른 예**: ` ```mermaid ... ``` `  
  - **잘못된 예**: `{% highlight text %} ```mermaid ... ``` {% endhighlight %}`
- **Mermaid Front-matter**: 글 본문에 Mermaid 다이어그램이 포함될 경우, 상단 YAML Front-matter 영역에 반드시 `mermaid: true` 속성을 명시해야 렌더링된다.
- **Mermaid 노드 라벨 안전 규칙**: 노드 라벨에 `<br>` 태그를 사용하면 Chirpy에서 파싱 에러가 발생한다. 줄바꿈이 필요하면 라벨을 짧게 줄이거나 공백으로 구분한다. 공백이 포함된 라벨은 반드시 큰따옴표로 묶는다. (예: `Node["라벨 텍스트"]`)

### 8. [DevOps 포트폴리오 관점] 커리어 컨텍스트 반영 의무
- **사용자 커리어 방향**: 이 블로그의 주인은 DevOps/인프라 엔지니어를 목표로 한다.
  단순한 에러 해결 회고록이 아니라, 아래 질문에 답하는 글이어야 한다:
  "이 엔지니어는 인프라 장애가 났을 때 어떻게 생각하고 판단했는가?"
- **면접관 관점 필수 포함**: 모든 원고의 결론부에 반드시 아래 2가지 중
  하나 이상이 포함되어야 한다.
  1. FinOps 관점: "왜 더 비싼/저렴한 선택을 했는가?"
  2. SRE 관점: "이 결정이 장애 상황에서 어떻게 작동하는가?"
- **AI 코드 운영 경험 강조**: 블로그에 앱 코드가 등장할 경우,
  "코드를 짰다"가 아니라 "AI가 생성한 코드를 리뷰하고 프로덕션에서
  책임지고 운영했다"는 관점으로 서술한다.

### 9. [SEO 최적화] 포트폴리오 블로그 검색 노출 의무
- **Front-matter 필수 항목**: 모든 포스트에 반드시 아래를 포함한다.
  - `description`: 검색 결과에 노출될 160자 이내 요약 (핵심 키워드 포함)
  - `tags`: 핵심 기술 키워드 3~5개 (예: `[Terraform, ECS, Fargate, DevOps, AWS]`)
- **제목(title) 원칙**: 검색 가능한 기술 키워드가 반드시 1개 이상 포함되어야 한다.
  - 나쁜 예: "삽질의 기록", "드디어 해결했다"
  - 좋은 예: "Terraform으로 ECS Fargate 전환하며 겪은 State Drift 트러블슈팅"
- **시리즈 연결**: 2부작 이상의 글은 각 포스트 하단에 이전/다음 편 링크를 반드시 삽입하여
  검색 크롤러의 내부 링크 점수를 높인다.

### 10. [Pipeline Consumer] 블로그 큐 데이터 렌더링 자동화
- **[큐 스캔 및 소비]**: 사용자가 "블로그 렌더링" 또는 "원고 작성"을 지시하면, AI는 즉시 `C:\Users\user\Desktop\Blog_Queue` 폴더를 스캔하여 쌓여있는 `raw_journal_*.md` 파일(트러블슈팅 원시 데이터)을 읽어 들인다.
- **[SSG 포맷팅 변환]**: 읽어 들인 날것의 로그를 분석하여, 본 워크스페이스의 핵심 룰인 **제 5원칙(4단 구조)**과 **제 7원칙(Jekyll 코드 블록 규칙)**에 맞게 완벽한 포트폴리오용 마크다운 원고로 자동 변환(렌더링)한다.
- **[파일 발행 및 큐 소각]**: 변환이 완료된 원본 큐 파일(`raw_journal_*.md`)은 중복 발행을 막기 위해 `Blog_Queue` 폴더에서 즉시 삭제(소각)하여 파이프라인의 데이터 무결성을 유지한다.

### 11. [Jargon Check] 전문 용어 소크라테스 검증 프로세스 (2026-07-13 실전 검증 도입)
> 이 규칙은 "블로그 원고를 내 언어로 방어할 수 없으면 포트폴리오가 아니다"는 원칙에서 출발합니다.

#### Phase 1: 원고 작성 전 — 용어 파인만 설명 (Pre-Writing)
- **트리거**: 블로그 원고 초안 생성이 완료된 직후.
- **실행**: AI는 원고 내에 등장하는 **영어 전문 용어, DevOps/클라우드 특수 용어, 보안 용어** 목록을 자동으로 추출한다.
- **파인만 비유 선제 제공**: 각 용어에 대해 이모지와 일상 언어 비유를 1:1로 매칭한 **파인만 비유 설명**을 제공한다. (예: SSL Termination = 안내데스크가 암호화된 택배를 대신 열어주는 것)
- **목표**: 사용자가 "아, 이게 이런 거구나!"를 느낀 뒤 원고를 읽을 수 있도록 사전 지식을 장착시킨다.

#### Phase 2: 원고 작성 중 — 한국어 병기 의무화 (In-Writing)
- **실행**: 원고 내 모든 영어 전문 용어 뒤에 `(한국어 직관 설명)` 형태로 괄호 설명을 필수로 삽입한다.
  - 올바른 예: `Blast Radius(보안 사고 시 피해가 연쇄적으로 퍼지는 범위)`
  - 올바른 예: `Ephemeral 러너(작업마다 텅 빈 새 컴퓨터로 시작하는 무상태 실행 환경)`
  - 잘못된 예: `Blast Radius` (설명 없이 영어만 단독 사용)
- **원칙**: SEO와 포트폴리오 키워드 가치를 위해 영어 용어는 유지하되, 독자(=면접관)가 용어를 모르더라도 맥락을 이해할 수 있도록 한다.

#### Phase 3: 원고 작성 후 — 소크라테스 이해도 검증 (Post-Writing)
- **트리거**: 블로그 원고 초안이 완성되어 사용자에게 제시된 직후.
- **실행**: AI는 원고에서 추출한 핵심 전문 용어 중 **3~5개**를 선정하여 사용자에게 제시한다.
- **검증 방식**: 사용자가 직접 **"면접관 앞에서 말하듯"** 본인 언어로 설명하도록 요청한다. AI는 절대 정답을 먼저 제공하지 않는다 (제 1원칙 준수).
- **채점 기준**:
  - ✅ 80점 이상: "방어 가능" 판정, git push 승인
  - 🔴 80점 미만: 추가 소크라테스 질문으로 오개념 교정 후 재검증
- **Mercy Rule**: 같은 용어로 3번 이상 막히면 직접 설명 제공 (글로벌 룰 제 3원칙 준수)
