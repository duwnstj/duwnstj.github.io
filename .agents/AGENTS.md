# 🧠 Workspace-Scoped Rules (블로그 프로젝트 전용)

> 이 워크스페이스는 기술 블로그(GitHub Pages)를 관리하고 마크다운(Markdown) 원고를 렌더링/업데이트하는 전용 공간입니다. AI는 아래의 규칙을 엄격하게 준수하여 원고의 퀄리티를 유지해야 합니다.

## 📌 [블로그 에디터 핵심 원칙]

### 1. 마크다운 및 Front-matter 표준화
- 블로그 포스트 작성 시 반드시 지킬 것: 포스트 상단에 블로그 프레임워크(Jekyll/Hugo/Next.js 등) 형식에 맞는 올바른 YAML Front-matter(제목, 날짜, 카테고리, 태그 등)를 정확하게 삽입한다.
- 모든 원고는 한국어(Korean)로 작성하며, 기술 용어는 영어 표기를 병기하여 전문성을 높인다. (예: 의존성(Dependency), 무중단 배포(Zero-Downtime Deployment))

### 2. 다이어그램(Mermaid) 및 퀄리티 우선주의 (No Image Capture)
- **이미지 캡처 전면 금지**: 블로그 원고 내에 터미널 에러나 단순 콘솔 화면 캡처 이미지를 첨부하는 행위를 완전히 배제한다. 
- **직관적인 시스템 다이어그램**: 복잡한 인프라 구조, 망 분리 아키텍처, 혹은 트러블슈팅 파이프라인은 반드시 **Mermaid 다이어그램**을 사용하여 구조적이고 직관적으로 시각화한다.

### 3. 서사(Storytelling) 중심의 편집 (Aha-Moment 보존)
- 단순한 코드 나열, 뻔한 성공담, 영혼 없는 기계적 요약은 엄격히 배제한다.
- **CRITICAL**: "어떤 에러가 발생했는가 ➔ 왜 발생했는가 ➔ 어떤 비유를 통해 아키텍처를 통찰했는가 ➔ 어떻게 해결했는가"라는 본질적인 사고의 흐름과 텍스트 퀄리티 자체를 유지한다.
- 원고 내에 사용자의 깨달음(Aha-Moment, 예: "아! 이게 이런 구조였구나!")이나 AI와의 대화형 티키타카 서사가 포함되어 있다면, 이를 절대 훼손하거나 요약하지 않고 '날것 그대로의 생생한 대화체'로 블로그 원고에 반영한다.

### 4. 구조적이고 가독성 높은 포맷팅
- 긴 텍스트는 GitHub Alerts(`> [!NOTE]`, `> [!WARNING]` 등)와 콜아웃 박스를 활용하여 가독성을 극대화한다.
- 면접관이나 독자가 읽었을 때 기술적 깊이를 느낄 수 있도록, Trade-off(의사결정 이유)와 SRE/FinOps 관점에서의 분석을 글의 결론부에 반드시 포함한다.
